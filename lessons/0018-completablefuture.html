CompletableFuture 异步编排与组合

开篇连接

L06 你学会了自定义 ThreadPoolExecutor 和 CallerRunsPolicy 背压——但只是"把任务丢进去"，没解决"任务之间有依赖、要并行 fan-out 再聚合 fan-in"。这一课用 CompletableFuture 补上编排这一层：把"串行阻塞 future.get()"重构成"并行 + 依赖链 + 异常兜底 + 超时"。面试高频三连——thenApply 与 thenCompose 区别 / 回调在哪个线程跑 / 多个 RPC 怎么并行——学完都能现场推导。落地锚点是你 GitHub 第 5 项 batch-import-optimizer：把 10 万条数据"分批 → 并行处理 → 聚合 → 收集失败批次"，以及 clickhouse-mcp-server 多分片查询的并行扇出。

前置检查

前置 1（来自 L06）：Future.get() 为什么阻塞？三个 Future 依次 get() 的性能问题在哪？

点开核对

get() 会阻塞调用线程直到结果就绪——它是拉模型（pull），你主动去等。三个 Future 依次 get，虽然任务在线程池里可能已并行提交，但你的取值是串行等待：总耗时 ≈ max 还是 sum？取决于你提交顺序，但只要在 get(A) 没返回前不去 get(B)，就有"前一个没到、后一个干等"的窗口，且无法在结果就绪时回调触发下一步——只能轮询或阻塞。CompletableFuture 是推模型（push）：完成时主动回调，天然支持依赖编排。

前置 2（来自 L06）：自定义 ThreadPoolExecutor 的核心 7 参数是什么？为什么 IO 密集任务不能用 ForkJoinPool.commonPool()？

点开核对

7 参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler（拒绝策略）。commonPool 默认并行度 = CPU 核数 - 1，是为CPU 密集、短任务、work-stealing 设计的；IO 密集任务会长时间占住线程，几个阻塞调用就能耗尽公共池，导致全 JVM 共享 commonPool 的所有并行流 / CompletableFuture 连锁卡死。所以 IO 场景必须传自定义 Executor。

前置 3（来自 L17）：volatile 如何保证一写一读的可见性？这与"异步回调能读到任务结果"有什么关系？

点开核对

volatile 写插 StoreLoad 屏障刷出 store buffer，volatile 读使缓存失效从主存读，建立 写 happens-before 读。CompletableFuture 内部用 volatile result 字段 + CAS：任务线程写 result（带 volatile 语义），回调线程读 result——任务完成 happens-before 回调执行，所以回调里能安全读到结果，哪怕跨了线程池线程。这正是本课第 9 节要回贴 L17 的点。

1. 创建：supplyAsync / runAsync 与线程池

四个入口方法，区别只在"有无返回值"和"用哪个线程池"：

方法返回线程池
supplyAsync(Supplier)有结果 CF<T>commonPool
supplyAsync(Supplier, executor)有结果 CF<T>自定义
runAsync(Runnable)无结果 CF<Void>commonPool
runAsync(Runnable, executor)无结果 CF<Void>自定义
private static final ExecutorService POOL = new ThreadPoolExecutor(
    8, 16, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(200),
    new ThreadFactoryBuilder().setNameFormat("ch-query-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()); // L06 背压

CompletableFuture<String> cf = CompletableFuture.supplyAsync(
    () -> queryShard("shard-1"), POOL);  // 始终传自定义池！

第一性原则：只要任务里有 IO（DB / RPC / HTTP），永远显式传 Executor。不传 = 用 commonPool = 和全 JVM 所有人抢 CPU-1 个线程。

2. 变换链：thenApply vs thenCompose vs thenCombine

心智模型直接对标函数式：thenApply = map，thenCompose = flatMap，thenCombine = zip。

方法入参函数签名用途类比
thenApplyT -> U同步映射结果map
thenComposeT -> CF<U>串接另一个异步调用，平铺flatMap
thenCombine(T,U) -> V合并两个独立 CF 的结果zip
thenAcceptT -> void消费结果，无返回forEach
thenRun() -> void不关心结果，跑个收尾—

关键区别：当下一步本身返回 CompletableFuture 时，用 thenApply 会得到嵌套 CF<CF<U>>，必须用 thenCompose 平铺。

// getUser 返回 CF<User>，getOrders 返回 CF<List<Order>>

// ❌ 嵌套地狱：CF<CF<List<Order>>>
CompletableFuture<CompletableFuture<List<Order>>> bad =
    getUser(id).thenApply(u -> getOrders(u.getId()));

// ✅ 平铺：CF<List<Order>>
CompletableFuture<List<Order>> good =
    getUser(id).thenCompose(u -> getOrders(u.getId()));

// thenApply 用于纯同步变换（无新异步调用）
CompletableFuture<String> name =
    getUser(id).thenApply(User::getName);  // User -> String，map

即时检验：queryUser(id) 返回 CF<User>，queryProfile(user) 返回 CF<Profile>。要把 user 串到 profile，用哪个方法？为什么不能用 thenApply？

核对

thenCompose。因为 queryProfile 返回的是 CF<Profile>，thenApply 会包成 CF<CF<Profile>>，需要 flatMap 语义平铺成 CF<Profile>。

3. *Async 后缀与线程归属

这是面试"哪个线程执行"的核心。每个变换方法都有三个重载，决定回调跑在哪：

形式回调执行线程
thenApply(fn)（无 Async）谁完成上游就用谁的线程；若上游已完成，则用调用者线程（同步执行）
thenApplyAsync(fn)提交到 commonPool
thenApplyAsync(fn, executor)提交到指定 Executor
CompletableFuture.supplyAsync(() -> step1(), POOL)   // 在 POOL 线程跑
    .thenApply(r -> step2(r))                    // 通常仍在 POOL 线程（沿用上游）
    .thenApplyAsync(r -> step3(r))               // 跳到 commonPool！
    .thenApplyAsync(r -> step4(r), POOL);          // 跳回 POOL

判定口诀：无 Async → 跟上游线程（或调用线程，若上游已就绪）；带 Async 无 executor → commonPool；带 Async + executor → 你指定的池。链式编排里如果每一步都要可控线程，统一用 xxxAsync(fn, POOL)，别混。

易错点：thenApply（无 Async）不保证不在调用线程跑。当上游 future 此刻已经完成，回调会同步在当前线程立即执行——这就是为什么"我以为回调异步、结果阻塞了主线程"的坑。

即时检验：supplyAsync(task, POOL).thenApply(f1).thenApplyAsync(f2)，假设 task 耗时未完成。f1 和 f2 分别在哪个线程？

核对

f1 在 POOL 线程（沿用上游完成线程，无 Async）；f2 在 commonPool（thenApplyAsync 未传 executor）。

4. 组合：thenCombine / allOf / anyOf

thenCombine — 两个独立任务并行后合并

CompletableFuture<Integer> a = CompletableFuture.supplyAsync(() -> countA(), POOL);
CompletableFuture<Integer> b = CompletableFuture.supplyAsync(() -> countB(), POOL);
CompletableFuture<Integer> sum = a.thenCombine(b, (x, y) -> x + y);
// a、b 并行跑，都完成后合并；总耗时 ≈ max(a,b)，不是 a+b

allOf — fan-in，等全部完成

List<CompletableFuture<Row>> futures = shards.stream()
    .map(s -> CompletableFuture.supplyAsync(() -> queryShard(s), POOL))
    .collect(Collectors.toList());

CompletableFuture<Void> all = CompletableFuture.allOf(
    futures.toArray(new CompletableFuture[0]));

// allOf 返回 CF<Void>，拿不到结果——完成后再 join 各子 future 取值
List<Row> rows = all.thenApply(v ->
    futures.stream().map(CompletableFuture::join).collect(Collectors.toList())
).join();

allOf 的返回是 CF<Void>，不携带任何子结果。范式是：allOf 完成后，在 thenApply 里对原始 future 列表逐个 join()（此时它们都已完成，join 不阻塞）取值聚合。

anyOf — 谁先完成用谁（如多副本取最快）

// 回调线程归属：allOf/anyOf 的后续回调，跑在「最后/最先完成那个任务的线程」上（无 Async 时）
CompletableFuture<Object> fastest = CompletableFuture.anyOf(replica1, replica2, replica3);

5. 异常处理三件套：exceptionally / handle / whenComplete

方法上游正常上游异常能改结果？函数签名
exceptionally不执行（透传原值）执行能（提供兜底值）Throwable -> T
handle执行执行能（正常+异常都能改）(T, Throwable) -> U
whenComplete执行执行不能（只观察，原值/原异常透传）(T, Throwable) -> void
queryShard(s)
    .exceptionally(ex -> {                  // 仅异常路径：返回兜底空行
        log.warn("shard {} failed", s, ex);
        return Row.EMPTY;
    })
    .whenComplete((row, ex) -> {            // 观察：埋点/释放资源，不改结果
        meter.record(s, ex == null);
    });

// handle：无论成败都收口，常用于「成功取值 / 失败转默认」二合一
queryShard(s).handle((row, ex) -> ex == null ? row : Row.EMPTY);

异常沿链短路：链中某步抛异常，后续所有 thenApply / thenCompose 都被跳过，异常一路传到第一个 exceptionally / handle。这和 try-catch 不同——它是数据流式的短路传播，包装成 CompletionException，记得 getCause() 拆包。

即时检验：上游正常完成时，exceptionally / handle / whenComplete 各会不会执行？

核对

exceptionally 不执行（透传正常值）；handle 执行（Throwable 为 null）；whenComplete 执行（Throwable 为 null，但不能改结果）。

6. 超时控制（Java 9+）

方法（JDK9+）超时行为
orTimeout(t, unit)超时则以 TimeoutException 异常完成
completeOnTimeout(val, t, unit)超时则以默认值 val 正常完成（降级兜底）
queryShard(s)
    .orTimeout(2, TimeUnit.SECONDS)              // 2s 没回 → TimeoutException
    .exceptionally(ex -> Row.EMPTY);            // 配合兜底

// 或直接超时返回默认值（不抛异常）
queryShard(s).completeOnTimeout(Row.EMPTY, 2, TimeUnit.SECONDS);

JDK8 没有这两个方法。替代方案：自建一个 ScheduledExecutorService，到点调用 future.completeExceptionally(new TimeoutException())，用 acceptEither 或谁先完成谁生效来实现超时。注意：orTimeout 内部用的是公共的 Delayer 单线程调度器，超时回调跑在它上面，别在超时分支做重活。

7. 实战：并行查询多个 ClickHouse 分片 + batch-import 编排

把前面所有零件拼成生产级 fan-out / fan-in：并行扇出 → 单分片超时 + 异常兜底 → allOf 聚合 → 收集失败分片。

public AggResult queryAllShards(List<String> shards, String sql) {
    Map<String, Throwable> failures = new ConcurrentHashMap<>();

    List<CompletableFuture<Row>> futures = shards.stream()
        .map(s -> CompletableFuture
            .supplyAsync(() -> client.query(s, sql), POOL)   // 自定义池
            .orTimeout(3, TimeUnit.SECONDS)                  // 单片超时
            .exceptionally(ex -> {                          // 失败不拖垮整体
                failures.put(s, ex);
                return Row.EMPTY;
            }))
        .collect(Collectors.toList());

    List<Row> rows = CompletableFuture
        .allOf(futures.toArray(new CompletableFuture[0]))
        .thenApply(v -> futures.stream()
            .map(CompletableFuture::join)
            .collect(Collectors.toList()))
        .join();

    return new AggResult(merge(rows), failures);  // 聚合 + 失败清单可重试
}

batch-import-optimizer 变体：把 10 万条数据按 1000 条切批，每批一个 supplyAsync(() -> insertBatch(chunk), POOL)，用 L06 的 Semaphore 限流 + CallerRunsPolicy 背压控制在途批数，allOf 聚合，exceptionally 把失败批次 chunkId 收进并发集合，最后对失败批次做有限次重试。这就是"分批 → 并行 → 聚合 → 重试"的完整骨架。

8. 陷阱清单（面试高频追问）

  1. thenApply / thenApplyAsync 线程混淆：无 Async 不一定异步，上游已完成时会同步在调用线程跑，可能阻塞调用方。
  2. commonPool 并行度 = CPU - 1：4 核机器只有 3 个线程，几个 IO 阻塞任务就耗尽，且全 JVM 共享 → 连锁卡死。IO 任务必须传自定义池。
  3. cancel() 不中断底层任务：cf.cancel(true) 只把 future 标记为异常完成，不会真正中断已在跑的 supplyAsync 任务（mayInterruptIfRunning 对它无效），任务还在线程池里跑完。要可中断需自己持有 Thread/Future 并处理 interrupt。
  4. 异常被吞：链尾不接 exceptionally/handle，异常默默存在 future 里，不 join/get 就永远看不到——容易"静默失败"。
  5. ThreadLocal / MDC 上下文丢失：thenXxxAsync 切线程后，ThreadLocal、日志 MDC traceId、事务上下文全部失效（伏笔 L07，解法 TransmittableThreadLocal 或装饰 Executor 透传上下文）。
  6. join() vs get()：join 抛非受检 CompletionException，get 抛受检 ExecutionException + InterruptedException；流式编排里用 join 更顺手。

9. 横向对照与跨课串联

跨课底座：

  • L06：CompletableFuture 必须配自定义 Executor，否则掉进 commonPool 陷阱——编排是"上层骨架"，线程池是"下层引擎"。
  • L14（AQS/CAS）：CompletableFuture 内部把待触发的回调串成 Completion 链表（Treiber 栈），用 CAS 无锁压栈/触发；commonPool 即 ForkJoinPool 的 work-stealing 队列也是 CAS 驱动。
  • L17（JMM）：回调能读到任务结果，靠的是 result 字段 volatile 写 happens-before 回调中的读——任务完成 hb 回调执行，跨线程可见性由此保证。
  • L07（ThreadLocal）：异步切线程后 ThreadLocal 失效，引出 TransmittableThreadLocal 与上下文透传。

面试话术（项目锚定）

Q：你怎么把多个 RPC/DB 调用并行化？

"早期我们是串行 future.get()，三个分片查询要等三次，总耗时是累加。我重构成 CompletableFuture：每个分片 supplyAsync 提交到自定义线程池同时跑，用 allOf 等全部完成，再在回调里逐个 join 聚合，总耗时从三段之和降到最慢一段。在 clickhouse-mcp-server 里这就是 MCP 工具内部的并行扇出优化。"

Q：thenApply 和 thenCompose 区别？

"对应 map 和 flatMap。thenApply 接受 T->U 做纯同步变换；当下一步本身返回 CompletableFuture 时，用 thenApply 会得到嵌套的 CF<CF<U>>，必须用 thenCompose 平铺成 CF<U>。判断标准就一句：下一步是不是又一个异步调用，是就 thenCompose。"

Q：回调到底在哪个线程跑？

"分三种：无 Async 后缀沿用上游完成线程，但上游若已完成则在调用线程同步执行——这是个坑，可能阻塞调用方；带 Async 不传 executor 跑在 commonPool；带 Async 传 executor 跑在指定池。生产里我统一用 xxxAsync(fn, 自定义池)，避免 IO 任务掉进 commonPool——它默认并行度只有 CPU-1，且全 JVM 共享，IO 阻塞会连锁卡死。"

Q：异步任务抛异常怎么传播？

"异常沿链短路：某步抛异常后续 then 全跳过，传到第一个 exceptionally 或 handle。我用 exceptionally 做仅异常路径的兜底降级，用 handle 做正常异常都收口的二合一，用 whenComplete 做只观察不改结果的埋点和资源释放。再叠 orTimeout 做单任务超时，失败分片收进并发集合后做有限重试——这套就是 batch-import-optimizer 可控批量导入的容错骨架。"

Q（追问底层）：CompletableFuture 完成后怎么触发那么多回调的？

"它内部把每个等待回调封装成 Completion 节点，串成一个 Treiber 栈（无锁链表），注册回调时用 CAS 压栈。任务完成时 CAS 写入 volatile 的 result 字段，然后弹栈依次触发。CAS 接 L14 的无锁机制，volatile result 的写 happens-before 回调读接 L17,保证跨线程可见;而 commonPool 即 ForkJoinPool 的 work-stealing 也是 CAS 驱动的。"

课后首要行动

  1. 写并对比线程归属：起一段 supplyAsync(...).thenApply(...).thenApplyAsync(...).thenApplyAsync(..., pool)，每步打印 Thread.currentThread().getName()，亲眼看 commonPool 与自定义池的切换。
  2. 复现 commonPool 卡死：用 commonPool 提交 50 个各 sleep 2s 的"IO 任务"，观察并行度只有 CPU-1、其余排队；换成自定义 16 线程池对比吞吐。
  3. 给 batch-import-optimizer 落地编排：把"分批 → supplyAsync 并行插入（自定义池 + Semaphore 限流）→ allOf 聚合 → exceptionally 收集失败批次 → 重试"骨架写进项目，提交一次 commit。
