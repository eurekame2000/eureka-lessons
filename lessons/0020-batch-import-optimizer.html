1. 前置检查（30 秒自测）

这四课是本项目的地基。先确认你没漏：

Q-pre1：线程池任务提交的完整顺序是什么？（L06）

核心线程 → 有界队列 → 创建非核心线程（到 maxPoolSize）→ 触发拒绝策略。注意顺序：队列在非核心线程之前，这是 JDK 线程池和直觉相反的关键点。

Q-pre2：为什么「无界队列 + 固定线程池」会 OOM，且永远不触发拒绝策略？（L06）

无界队列（LinkedBlockingQueue 无容量参数）永远「装得下」，所以线程数永远不会扩到 maxPoolSize，拒绝策略永不触发。任务无限堆积在堆内存里 → OOM。Executors.newFixedThreadPool 正是这个坑。

Q-pre3：allOf 之后怎么取各子结果且不阻塞？（L18）

CompletableFuture.allOf(f1, f2, f3).join() 之后，所有子 future 已完成，再对每个调 .join() 立刻返回、不阻塞。

Q-pre4：TtlExecutors.getTtlExecutorService 解决什么问题？（L19）

线程池复用线程，打断 ThreadLocal/MDC 继承链。包装一次 Executor，后续所有 *Async 提交自动在「提交时拍快照、执行时还原」，保证 traceId 不丢。

以上有任何一题卡壳，先回去复习对应课，再继续——本课不重讲理论，只讲怎么把它们拼起来。

2. 项目定位与 API 设计

batch-import-optimizer 是一个通用、零依赖框架的 Java 受控并发批量导入库。给它一个大集合和一个「单批写入函数」，它负责分批、限流、并发、背压、失败收集，调用方只关心业务写入逻辑。

2.1 三个核心类

类职责用到的课
BatchImportConfig不可变配置：批大小、并发度、下游许可数、重试次数、超时—
BatchImportExecutor<T>核心引擎：建有界线程池、Semaphore、分批 fan-out/fan-in、收集结果L06+L14+L18+L19
BatchImportResult结果聚合：成功批数、失败批次清单（含原始数据，供重试）L18 handle

2.2 调用方视角（先看用法再看实现）

// 调用方只写这几行
BatchImportConfig config = BatchImportConfig.builder()
        .batchSize(500)           // 每批 500 条
        .concurrency(8)           // 最多 8 个批次并行（线程池 maxPoolSize）
        .downstreamPermits(4)     // 下游 DB 同时最多 4 个连接（Semaphore）

// 成功 1998 批 / 失败 2 批，失败数据已在 result.failedBatches() 里

3. 受控并发核心：有界队列 + CallerRunsPolicy

最关键的一行设计：绝不用 Executors 工厂方法，手动 new 一个带有界 ArrayBlockingQueue的 ThreadPoolExecutor，拒绝策略选 CallerRunsPolicy。 

private ThreadPoolExecutor buildPool(BatchImportConfig cfg) {
    int core = cfg.getConcurrency();
    return new ThreadPoolExecutor(
            core,                                  // corePoolSize
            core,                                  // maxPoolSize（导入是 IO 密集，core==max 简化）
            60L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(core * 2),         // ★ 有界队列：缓冲但不无限堆积
            new ThreadFactoryBuilder()
                    .setNameFormat("batch-import-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy()  // ★ 背压核心
    );
}
生产者(主线程) 有界队列(cap=16) worker线程×8 │ │ │ │ 提交 batch ──────────▶ [■■■■■■■■] │ 并发执行 │ │ 满了！ │ │ 提交 batch ──────╳───▶ [拒绝] │ │ │ ▼ CallerRunsPolicy: 主线程自己跑这个 batch │ 主线程被「占用」执行任务 → 暂停提交 → 天然降速

3.1 为什么 CallerRunsPolicy 能形成背压

当队列和线程都满，新任务被拒。CallerRunsPolicy 不抛异常、不丢弃，而是让提交任务的那个线程（caller，这里是主线程）自己同步执行这个任务。主线程在跑任务期间无法继续 submit，于是生产速度自动被「下游处理速度」拖住——这就是背压（backpressure）：快的一方被慢的一方反向限速。

拒绝策略行为代价
AbortPolicy（默认）抛 RejectedExecutionException任务丢失，需调用方 catch 重试，复杂
DiscardPolicy静默丢弃数据丢失且无感知 —— 批量导入绝对禁用
DiscardOldestPolicy丢队首旧任务丢的是先提交的批次，更不可接受
CallerRunsPolicy调用方线程自己执行不丢数据 + 自动降速；代价是主线程偶尔被占用、单批延迟上升
面试金句：「CallerRunsPolicy 用『让生产者干活』换『不丢数据 + 自动反压』。对吞吐：稳态吞吐由下游能力决定，不会被生产端冲垮；对延迟：被回退的那批由主线程串行执行，单批延迟升高，但系统整体不雪崩。」
即时检验：core=4, max=4, 队列 cap=8，一次性提交 20 个 batch，会发生什么？

4 个直接被 worker 执行，8 个进队列，剩下 8 个触发拒绝 → CallerRunsPolicy 让主线程逐个同步执行这 8 个。整个过程无异常、无丢失，主线程被反压住，提交节奏自动匹配消费速度。

4. 两层限流：线程池并发 ≠ 下游并发

线程池的 maxPoolSize 限的是「计算侧并发」——同时有几个批次在被处理。但每个批次内部可能调用 DB / 第三方接口，真正要保护的是「下游连接并发」。两者语义不同，复杂场景需要叠加。

线程池 maxPoolSizeSemaphore 许可
限制对象同时运行的任务数同时持有许可的代码段数
粒度整个任务生命周期可只包住下游调用那一小段
底层线程数 + 队列AQS 共享模式（L14）
典型用途控总并发、控内存保护 DB 连接池 / 第三方 QPS
private final Semaphore downstreamLimit;  // = new Semaphore(cfg.getDownstreamPermits())

private BatchOutcome runBatch(List<T> batch, BatchWriter<T> writer) {
    try {
        downstreamLimit.acquire();          // ★ 进入下游前申请许可（AQS 共享锁）
        try {
            writer.write(batch);            // 真正打 DB 的那一段
            return BatchOutcome.success(batch.size());
        } finally {
            downstreamLimit.release();      // ★ 必须 finally 释放，否则许可泄漏
        }
    } catch (Exception e) {
        return BatchOutcome.failure(batch, e); // 异常隔离：不抛出，转成结果
    }
}
为什么要两层？假设线程池 8 并发，但 DB 连接池只有 4。若不加 Semaphore，8 个批次同时抢 4 个连接，4 个线程阻塞在连接获取上、白白占着线程。用 Semaphore(4) 包住下游调用，让多余批次在申请许可时等待，线程行为更可控、超时更好管理。

5. CompletableFuture 编排 + 失败收集

分片 → 每片 supplyAsync 到自定义池 → handle 把异常转成结果（不让异常炸穿）→ allOf fan-in → 汇总。每批异常独立隔离，第 3 批失败不影响第 4 批。 

public BatchImportResult importAll(List<T> all, BatchWriter<T> writer) {
    List<List<T>> batches = partition(all, config.getBatchSize());

    List<CompletableFuture<BatchOutcome>> futures = batches.stream()
        .map(batch -> CompletableFuture
            .supplyAsync(() -> runBatch(batch, writer), pool)  // pool 已被 TtlExecutors 包装
            .handle((outcome, ex) -> {                          // ★ handle 同时拿正常值和异常
                if (ex != null) return BatchOutcome.failure(batch, ex);
                return outcome;
            }))
        .collect(Collectors.toList());

    // fan-in：等全部完成（无论成败），再逐个取结果
    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();

    List<BatchOutcome> outcomes = futures.stream()
            .map(CompletableFuture::join)   // allOf 后 join 不阻塞
            .collect(Collectors.toList());

    return BatchImportResult.from(outcomes);  // 聚合成功/失败
}
handle vs exceptionally：用 handle 而非 exceptionally，因为 handle 一个回调同时处理「正常结果」和「异常」，把两条路统一成 BatchOutcome，下游 allOf 永远不会因某批异常而整体失败。这是「失败隔离」的关键技巧。

5.1 失败批次选择性重试

BatchImportResult result = executor.importAll(all, writer);
if (result.hasFailures() && config.getMaxRetries() > 0) {
    List<T> retryData = result.failedBatches().stream()
            .flatMap(b -> b.data().stream())
            .collect(Collectors.toList());
    // 只重试失败的数据，且要求 writer 幂等（按主键 upsert）
    result = executor.importAll(retryData, writer);
}
重试前提是幂等。批量写必须用 INSERT ... ON DUPLICATE KEY UPDATE / upsert / 主键去重，否则重试会产生重复数据。这点写进 README 的「使用约束」。

6. 链路透传：让批任务日志带上 traceId

批任务跑在线程池里，MDC.get("traceId") 会是 null（L19 结论）。修复只需一行——用 TtlExecutors 包装线程池： 

import com.alibaba.ttl.threadpool.TtlExecutors;
import com.alibaba.ttl.TransmittableThreadLocal;

// traceId 放进 TTL（而非普通 ThreadLocal）
private static final TransmittableThreadLocal<String> TRACE = new TransmittableThreadLocal<>();

// 构造引擎时包装池：改这一行，所有 supplyAsync 自动透传
this.pool = TtlExecutors.getTtlExecutorService(buildPool(config));

这样每个批次 worker 执行前会 replay 主线程的 TTL 快照，配合 MDC Transmittee，日志里每条批写入记录都带 traceId，排查「哪一批慢、哪一批失败」时一搜即得。

即时检验：如果忘了包装、直接把裸 ThreadPoolExecutor 传给 supplyAsync，会怎样？

批任务里 MDC.get("traceId") 返回 null（或上一个复用任务的脏值），日志无法关联到本次请求链路。修复：TtlExecutors.getTtlExecutorService(pool) 包一层即可，业务代码零改动。

7. README 设计思路（讲清「为什么这么选」）

GitHub 项目的价值一半在 README。它要回答面试官会追问的三个「为什么」：

  • 为什么不用 Executors.newFixedThreadPool？ 它内部是无界 LinkedBlockingQueue，高压下任务无限堆积 → OOM，且拒绝策略永不触发。手动 new 才能控队列容量。
  • 为什么选 CallerRunsPolicy 而非默认 AbortPolicy？ 批量导入要的是「全量写完、一条不丢」，CallerRunsPolicy 提供天然背压且零数据丢失，比上层 catch-retry 简单可靠。
  • 为什么线程池之外还要 Semaphore？ 区分计算并发与下游连接并发，独立保护 DB，避免线程阻塞在连接获取上。
README 应包含的小节：① 解决什么问题（OOM / 丢数据痛点）② 快速开始（10 行 demo）③ 配置项表格 ④ 设计决策（上面三个为什么）⑤ 基准数据（背压前后吞吐对比）⑥ 使用约束（幂等要求）。

7.1 单元测试验证背压

// 用 CountDownLatch + AtomicInteger 验证「同时运行的批次数」不超 concurrency
@Test
void shouldNotExceedConcurrency() throws Exception {
    int concurrency = 4;
    AtomicInteger running = new AtomicInteger();
    AtomicInteger maxSeen = new AtomicInteger();
    BatchImportExecutor<Integer> ex = new BatchImportExecutor<>(
            BatchImportConfig.builder().batchSize(1).concurrency(concurrency).build());

    ex.importAll(range(0, 100), batch -> {
        int cur = running.incrementAndGet();
        maxSeen.updateAndGet(m -> Math.max(m, cur));
        Thread.sleep(10);
        running.decrementAndGet();
    });

    assertThat(maxSeen.get()).isLessThanOrEqualTo(concurrency);  // 并发被限住
}

8. 面试话术：STAR 项目叙事

Q：讲一个你做过的最有挑战的并发场景。

S（情境）：有个数据迁移任务，要把百万级用户从旧库批量导入新库。最初实现是 newFixedThreadPool + 来一批提交一批。
T（任务）：压测时主库被打挂，应用还偶发 OOM——线程池无界队列把待写批次全堆在堆里。
A（行动）：我抽了一个通用库 batch-import-optimizer。三件事：① 手动 new ThreadPoolExecutor 用有界 ArrayBlockingQueue，杜绝无限堆积；② 拒绝策略选 CallerRunsPolicy，队列满时让提交线程自己跑任务，形成背压，生产速度自动匹配下游；③ 再叠一层 Semaphore 限制对 DB 的真实并发连接数，区分计算并发与下游并发；编排上用 CompletableFuture 分批 fan-out/fan-in，handle 隔离每批异常、收集失败清单做选择性幂等重试；并用 TtlExecutors 包装线程池保证 traceId 在异步批任务里不断链。
R（结果）：内存平稳不再 OOM，DB 压力可控，吞吐稳定，失败批次可定位可重试。这个库我开源在 GitHub，核心抽象还复用进了另一个 MCP server 的多分片写入。
追问预案：
  • 「队列设满了怎么办？」→ 不会丢，CallerRunsPolicy 让 caller 执行，自动反压。
  • 「为什么不用无界队列？」→ OOM 根因 + 拒绝策略永不触发。
  • 「一批失败了怎么办？」→ handle 转结果不中断其他批 + 失败清单 + 幂等重试。
  • 「异步日志 traceId 丢了？」→ MDC 断链 + TtlExecutors 修复（回扣 L19）。

9. Quiz — 检索练习

每题只选一个最准确的答案。全对触发完成横幅。

Q1. core=3, max=3, 有界队列 cap=5，一次提交 12 个任务，多少个会触发拒绝策略？

✅ 正确！3 个直接执行 + 5 个入队 = 8 个被容纳，剩 4 个触发拒绝策略（这里被 CallerRunsPolicy 接管）。
❌ 顺序：先填 worker(3)，再填队列(5)，共容纳 8 个；12 - 8 = 4 个触发拒绝。

Q2. CallerRunsPolicy 形成背压，被拒任务由哪个线程执行？对系统有什么影响？

✅ 正确！caller（通常是主线程）同步执行被拒任务，期间无法继续 submit，生产速度被下游拖住——这就是背压。
❌ 核心是「caller runs」：提交线程自己跑，占用期间停止提交，形成反压。不丢数据。

Q3. 为什么 Executors.newFixedThreadPool 在高压批量导入下会 OOM？

✅ 正确！无界队列永远装得下，线程数不会扩、拒绝策略不触发，待处理任务全堆在堆里 → OOM。
❌ 根因是无界队列：任务无限入队、拒绝策略永不触发，堆积导致 OOM。所以本项目手动 new + 有界队列。

Q4. 线程池 maxPoolSize 和 Semaphore 许可都能限并发，什么场景必须同时用？

✅ 正确！maxPoolSize 限同时运行的任务，Semaphore 可只包住下游调用那一小段，独立保护 DB 连接池，语义不同。
❌ 二者语义不同：线程池限运行任务数，Semaphore 限下游调用段。当下游承受力低于计算并发时需叠加。

Q5. 分批任务中第 3 批抛异常，怎样保证其他批不中断且能定位重试失败批？

✅ 正确！handle 把异常路和正常路统一成结果对象，allOf 永不整体失败，失败批携带原数据进幂等重试。
❌ 关键是用 handle 做失败隔离：异常转 BatchOutcome.failure 携带原数据，allOf 不被某批异常炸穿，再收集重试。
