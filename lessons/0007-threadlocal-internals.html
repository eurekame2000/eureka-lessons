ThreadLocal 深度：线程池复用的定时炸弹

2026-06-20  ·  衔接 Lesson 06（线程池）+ Lesson 03（JVM/WeakReference） ·  预计 25 分钟
本课目标 1. 理解 ThreadLocal 的内部结构（ThreadLocalMap + WeakReference）
2. 能推导线程池复用下的脏数据 Bug 和内存泄漏路径
3. 掌握正确修复模式，背出面试标准答案

1. ThreadLocal 是什么——正确心智模型

ThreadLocal 不是"线程安全的共享变量"，而是每条线程独享一份变量副本。

Thread-A ──► ThreadLocalMap { k=threadLocal1 → v="user:alice" } Thread-B ──► ThreadLocalMap { k=threadLocal1 → v="user:bob" } ↑ 同一个 ThreadLocal 对象，不同 Map

每个 Thread 对象内部有一个字段 threadLocals，类型是 ThreadLocal.ThreadLocalMap。当你调用 threadLocal.set(val)，实际是：

// 伪代码，还原 JDK 实现
Thread t = Thread.currentThread();
ThreadLocalMap map = t.threadLocals;       // 拿到当前线程自己的 Map
if (map == null) map = createMap(t, value);
else map.set(this, value);                 // this = ThreadLocal 实例，作为 key

Entry 的结构（关键！）

// java.lang.ThreadLocal.ThreadLocalMap 内部类
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;                          // 强引用持有 value
    Entry(ThreadLocal<?> k, Object v) {
        super(k);                          // key 是弱引用
        value = v;                         // value 是强引用
    }
}
记忆口诀：key 弱，value 强 这个非对称设计是内存泄漏的根源——稍后展开。

2. 线程池复用 → 脏数据 Bug

线程池里的线程不会在任务结束后销毁，它会被复用处理下一个任务。

Bug 场景：用户 A 的数据出现在用户 B 的请求里 这是生产环境最隐蔽的数据串联 Bug，难以复现、难以定位。 
// 反例：不清理的 ThreadLocal
private static final ThreadLocal<String> USER_CTX = new ThreadLocal<>();

// Task 1：用户 alice 的请求
executor.submit(() -> {
    USER_CTX.set("alice");
    doWork();
    // 没有 remove()！线程归还给线程池
});

// Task 2：线程被复用，用户 bob 的请求
executor.submit(() -> {
    // USER_CTX.get() 返回 "alice" ←── 脏数据！
    String user = USER_CTX.get();
    doWork();
});

脏数据时序图

线程池 Thread-3： [Task-Alice] set("alice") → doWork → 归还（未清理） [Task-Bob] get() ──────────────────────────── 返回 "alice" ← BUG

3. 内存泄漏路径——画图推理

比脏数据更难发现的是内存泄漏。路径分两段：

阶段一：key 被 GC

GC Root │ └──► Thread（核心线程，线程池不销毁它） │ └──► threadLocals (ThreadLocalMap) │ └──► Entry[] ├── key: WeakReference<ThreadLocal> ──► ThreadLocal 实例 └── value: Object ──────────────────────────► [强引用，泄漏点] 当外部没有强引用指向 ThreadLocal 实例时： ThreadLocal 实例 ──── 仅剩 WeakReference ──── GC 可回收 → key 变为 null → Entry 变成 {key=null, value=残留对象} ← 无法被访问，也无法被 GC

阶段二：value 被强引用困住

Entry 里 key 是 null，但 value 是强引用，GC 无法回收 value（Thread 还活着 → ThreadLocalMap 活着 → Entry 活着 → value 活着）。

泄漏条件：三件事同时成立 ① 线程池核心线程长期存活（不销毁）
② ThreadLocal 对象没有外部强引用（或被置为 null）
③ 没有调用 remove()
JDK 的补丁式修复（了解即可） ThreadLocalMap 在 get()/set() 时会顺手清理 stale entry（key=null 的条目）。 但这是被动的，不能依赖它。主动 remove() 才是正解。

4. 正确修复模式

模式 A：try-finally remove（最通用）

private static final ThreadLocal<String> USER_CTX = new ThreadLocal<>();

public void handleRequest(String userId) {
    try {
        USER_CTX.set(userId);
        doWork();
    } finally {
        USER_CTX.remove();   // 无论异常与否，必须清理
    }
}
finally 块是唯一可靠位置 catch 块可能被跳过，在 try 末尾清理会因异常被绕过。

模式 B：封装成 AutoCloseable（更优雅）

public class UserContext implements AutoCloseable {
    private static final ThreadLocal<String> CTX = new ThreadLocal<>();

    public UserContext(String userId) { CTX.set(userId); }
    public static String get() { return CTX.get(); }

    @Override
    public void close() { CTX.remove(); }  // try-with-resources 自动调用
}

// 使用方
try (UserContext ctx = new UserContext(userId)) {
    doWork();  // 块内随时 UserContext.get()
}              // 离开块自动 remove()

模式 C：Spring 的 RequestContextHolder（框架层兜底）

Spring MVC 在 DispatcherServlet 里会在请求结束后调用 RequestContextHolder.resetRequestAttributes()，本质是清理线程本地的请求上下文。这是框架替你做了 remove()，但自定义 ThreadLocal 框架管不到。

5. 进阶：父子线程传递问题

InheritableThreadLocal — 父线程创建子线程时拷贝一次

private static final InheritableThreadLocal<String> TRACE =
    new InheritableThreadLocal<>();

TRACE.set("trace-id-001");
new Thread(() -> {
    System.out.println(TRACE.get()); // "trace-id-001" ← 继承自父线程
}).start();
InheritableThreadLocal 在线程池中失效 线程池里的线程是提前创建好的，不是在提交任务时新建，继承发生在线程创建时，不在任务提交时。 所以父线程提交任务后设置的值，线程池里的线程拿不到。

TransmittableThreadLocal (TTL) — 阿里开源，线程池正确透传

依赖：com.alibaba:transmittable-thread-local

private static final TransmittableThreadLocal<String> TRACE =
    new TransmittableThreadLocal<>();

// 提交任务时包装一层
executor.submit(TtlRunnable.get(() -> {
    System.out.println(TRACE.get()); // 正确拿到提交任务时的快照值
}));
使用场景 链路追踪（TraceId）、用户上下文（UserId/TenantId）在线程池异步任务中透传，用 TTL。 这是微服务链路追踪的标准方案。

6. 面试标准答案框架

面试题：ThreadLocal 内存泄漏的原因和解决方案？

结构：原理 → 根因 → 修复 → 延伸

  1. 原理：ThreadLocal 把数据存在 Thread 的 ThreadLocalMap 里，Entry 的 key 是 ThreadLocal 的弱引用，value 是强引用。
  2. 根因：线程池里线程不销毁；ThreadLocal 对象外部强引用消失后，key 被 GC 为 null；但 value 仍被 Entry 强引用，无法回收，随线程生命周期长期驻留堆内存。
  3. 修复：在 finally 块调用 remove()，或封装成 AutoCloseable。
  4. 延伸：如需父子线程透传，用 TTL（TransmittableThreadLocal），不要用 InheritableThreadLocal（线程池里失效）。
面试题：为什么 key 是弱引用，而不是强引用？

如果 key 是强引用，ThreadLocal 对象外部不持有引用时，ThreadLocalMap 仍强引用它，ThreadLocal 永远无法 GC —— 泄漏更严重。弱引用至少让 key 能被 GC 回收，是"两害取其轻"的设计。但 value 的泄漏问题无法靠弱引用解决，只能靠 remove()。
