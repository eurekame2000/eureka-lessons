Java 锁深度

synchronized 锁升级 vs ReentrantLock vs AQS 内部实现

Lesson 14 · 2026-06-20 · ⏱ 35 min · 进阶

开篇连接

面试里「synchronized 和 ReentrantLock 区别」是开场题，「AQS 是什么、ReentrantLock 怎么基于它实现」是中级 / 高级的分水岭。你已经在 L06 线程池 见过 Worker extends AQS，在 L12 见过 Redisson 分布式锁——这一课把 JVM 本地锁的底盘补齐，让你能用同一套「state + 等待队列 + CAS」的语言串起本地锁和分布式锁。

前置检查

前置 1：volatile 保证的两件事是什么？（关系到为什么 AQS 的 state 必须是 volatile）

查看答案

① 可见性：一个线程的写立即对其他线程可见（刷主存 + 失效其他 CPU 缓存行）；② 禁止重排序：volatile 写之前的操作不会重排到它后面，读之后的操作不会重排到它前面。注意：volatile 不保证原子性，所以 state++ 仍需 CAS。

前置 2（来自 L12）：Redisson 可重入锁靠什么记录重入次数？

查看答案

Redis Hash 里的 {uuid:threadId → count}，重入一次 count + 1，解锁一次 count - 1，归零才真正 DEL。记住这个 count——本课的 AQS state 在 ReentrantLock 里扮演的就是同一个角色。

1. synchronized 锁升级

很多人以为 synchronized 就是「重量级锁、慢」。JDK 6 之后它有了锁升级机制：根据竞争激烈程度，锁状态会单向升级，尽量避免操作系统级别的互斥开销。

Mark Word：锁状态存在哪里

每个 Java 对象头里有一个 Mark Word（64 位 JVM 下 64 bit）。它复用同一段空间，根据低位的锁标志位表达不同含义——这正是 L03 JVM 里讲对象头时埋下的伏笔。

状态Mark Word 主要内容锁标志位
无锁hashCode、分代年龄01 (biased=0)
偏向锁持有线程 ID、epoch、分代年龄01 (biased=1)
轻量级锁指向栈中 Lock Record 的指针00
重量级锁指向 monitor (ObjectMonitor) 的指针10

四阶段升级路径

无锁 ──首个线程进入──▸ 偏向锁  (CAS 把线程 ID 写进 Mark Word，之后同一线程无同步开销)
   │                      │
   │              第二个线程来竞争
   ▼                      ▼
   └──────────────▸ 轻量级锁 (CAS 自旋：把 Mark Word 换成指向栈 Lock Record 的指针)
                          │
                  自旋失败 / 竞争加剧
                          ▼
                     重量级锁 (膨胀为 ObjectMonitor，未抢到的线程被 OS 挂起 park)
阶段触发条件代价
偏向锁只有一个线程访问，无竞争近乎零（一次 CAS 后免同步）
轻量级锁出现第二个线程，但竞争不激烈（交替执行）CAS 自旋，消耗 CPU 但不进内核
重量级锁自旋达到阈值 / 多线程激烈争抢线程挂起 + 唤醒，涉及用户态↔内核态切换
单向升级，不会降级（运行期内）。一旦膨胀为重量级锁，这个对象就回不去了。所以「先用偏向 / 轻量级试探，扛不住再升级」是它的核心哲学。

JDK 15 为什么废弃偏向锁（JEP 374）

  • 收益下降：偏向锁是为「无竞争且重复加锁」场景设计的（如早期 Hashtable、StringBuffer）。现代代码大量使用 java.util.concurrent 而非 synchronized，偏向锁受益的场景越来越少。
  • 维护成本高：偏向锁的撤销（revoke）需要在安全点（safepoint）停止持有线程、修改 Mark Word，一旦发生竞争反而比直接用轻量级锁更慢，且代码极其复杂、难以维护。

JDK 15 默认关闭（-XX:+UseBiasedLocking 被标记 deprecated），后续版本逐步移除。面试答法：偏向锁不是「错」，是「投入产出比变低」——维护一段复杂的撤销逻辑去优化一个越来越少见的场景，不划算。

即时检验：第二个线程来竞争一把偏向锁时，锁会直接升级到重量级吗？

查看答案

不会。先撤销偏向 → 升级为轻量级锁（CAS 自旋）。只有自旋失败 / 竞争激烈才进一步膨胀为重量级锁。跳级是常见误区。

2. AQS 三大件

讲完 JVM 内置锁，进入 J.U.C 的发动机：AbstractQueuedSynchronizer (AQS)。ReentrantLock、CountDownLatch、Semaphore、ReentrantReadWriteLock、线程池的 Worker 全部建在它上面。

AQS 只有三个核心件：

public abstract class AbstractQueuedSynchronizer {
    // ① 同步状态：锁的"灵魂"，不同子类赋予不同语义
    private volatile int state;

    // ② CLH 变体双向队列的头尾指针：抢不到锁的线程在这里排队
    private transient volatile Node head;
    private transient volatile Node tail;

    // ③ CAS：所有对 state / 队列的修改都靠原子操作，无锁化
    protected final boolean compareAndSetState(int expect, int update) {
        return U.compareAndSetInt(this, STATE, expect, update);
    }
}

state 的语义复用——这是 AQS 最妙的设计

子类state 含义模式
ReentrantLock重入次数（0=未锁，n=重入 n 次）独占
CountDownLatch剩余计数（到 0 放行所有等待者）共享
Semaphore剩余许可数共享
ReentrantReadWriteLock高 16 位=读锁数，低 16 位=写锁重入共享+独占

Node 的 waitStatus

队列里每个 Node 包裹一个等待线程，waitStatus 描述它的状态：

SIGNAL    = -1;  // 后继节点需要被当前节点唤醒（释放时要 unpark 后继）
CANCELLED =  1;  // 线程被取消（超时/中断），需从队列移除
CONDITION = -2;  // 节点在 Condition 条件队列里等待
PROPAGATE = -3;  // 共享模式：释放需向后传播
一句话：AQS = 一个 volatile int（state）+ 一个 CAS 维护的双向等待队列（CLH 变体）。子类只需定义「state 怎么算获取成功」，排队 / 阻塞 / 唤醒全由 AQS 模板代劳。

3. AQS 模板方法骨架

AQS 用模板方法模式：骨架流程写死在父类，把「是否获取成功」的判断留给子类重写。

// 父类模板（独占获取，写死）
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&                                    // ① 子类实现：尝试抢
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))        // ② 抢不到 → 入队 → 阻塞
        selfInterrupt();
}

// 子类必须重写（父类默认抛异常）
protected boolean tryAcquire(int arg) {
    throw new UnsupportedOperationException();
}

acquire 的完整流程

acquire(1)
   │
   ▼ tryAcquire(1)  ── 子类逻辑：CAS state 0→1 成功? ──成功──▸ 拿到锁，返回
   │ 失败
   ▼ addWaiter(EXCLUSIVE)   把当前线程包成 Node，CAS 入队尾
   │
   ▼ acquireQueued(node, 1)
   │     循环：
   │       前驱是 head? ──是──▸ 再 tryAcquire 一次 ──成功──▸ 设自己为 head，返回
   │       否则 → 把前驱 waitStatus 设 SIGNAL → LockSupport.park(this)  ← 线程挂起
   ▼
release(1)
   ▼ tryRelease(1) ── 子类逻辑：state 减到 0? ──是──▸ unparkSuccessor(head)
                                                       LockSupport.unpark(后继线程)
方法归属职责
acquire / releaseAQS 父类模板：编排排队、阻塞、唤醒
tryAcquire / tryRelease子类重写定义 state 的获取 / 释放语义
addWaiter / acquireQueuedAQS 父类入队 + 自旋 + park
LockSupport.park/unpark底层线程阻塞 / 唤醒（基于 Unsafe）

对照 L12 Redisson：Redisson 的「抢不到锁就订阅 channel 等待唤醒」≈ AQS 的「入队 + park」；Redisson 解锁时 publish 通知 ≈ AQS 的 unparkSuccessor。一个在 Redis pub/sub 上排队，一个在 JVM 堆里的双向链表上排队，设计同构。

即时检验：tryAcquire 返回 false 后，线程是立刻 park 吗？

查看答案

不是。先 addWaiter 入队，然后在 acquireQueued 里自旋：如果发现自己前驱就是 head，还会再 tryAcquire 试一次（拿锁的可能性大）。只有确认前驱状态为 SIGNAL 后才真正 park。这是「先自旋探一下，再睡」的优化。

4. ReentrantLock：独占模式实战

可重入：state 累加 + exclusiveOwnerThread

// 非公平版 tryAcquire（简化）
protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {                                  // 锁空闲
        if (compareAndSetState(0, acquires)) {     // CAS 抢
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {  // 已是持有者 → 重入
        int nextc = c + acquires;                   // state 直接累加！
        setState(nextc);                            // 无需 CAS（独占，无竞争）
        return true;
    }
    return false;
}

解锁时 state 逐次减 1，减到 0 才真正释放并唤醒后继——这与 L12 Redisson 的 hash count 逻辑完全同构。

state 变化追踪（重要！quiz 必考）

lock();        // state: 0 → 1   (CAS 成功, owner=自己)
lock();        // state: 1 → 2   (重入, 直接 setState)
unlock();      // state: 2 → 1   (减 1, 未归零, 不释放)
unlock();      // state: 1 → 0   (归零, 清 owner, unpark 后继)
陷阱：加锁几次就要解锁几次。少调一次 unlock()，state 永远不为 0，锁永不释放 → 其他线程死等。所以 unlock() 必须放 finally。

公平 vs 非公平：就差一行

// 公平锁 FairSync.tryAcquire 比非公平多了这一行判断：
if (c == 0) {
    if (!hasQueuedPredecessors() &&         // ★ 关键：队列里有人排在我前面吗?
        compareAndSetState(0, acquires)) {
        setExclusiveOwnerThread(current);
        return true;
    }
}
非公平（默认）公平
新线程到来不看队列，直接 CAS 抢先查 hasQueuedPredecessors()，有人排队就乖乖入队
吞吐量高低
原因减少线程唤醒 / 上下文切换：刚释放锁时新线程"插队"成功，省去唤醒队首线程的开销严格 FIFO，但频繁 park/unpark 拖慢吞吐
风险队列线程可能饥饿无饥饿

为什么默认非公平：唤醒一个被 park 的线程要内核态切换，很慢。这段时间里如果有个新线程恰好来抢，让它"插队"直接拿走，整体吞吐更高。公平性的代价是吞吐——Doug Lea 默认选了吞吐。

synchronized vs ReentrantLock 总对比

维度synchronizedReentrantLock

Condition：条件队列 → 同步队列的转移

Condition 比 Object.wait/notify 强在「一把锁挂多个条件队列」（如生产者队列、消费者队列分开）。机制：

await()   线程从【同步队列】移出 → 包成 Node 挂入【条件队列】→ 释放锁 → park
            │
signal()  把【条件队列】队首 Node 转移回【同步队列】尾部
            │
            ▼ 它在同步队列里重新排队竞争锁，抢到才从 await() 返回

5. 共享模式：CountDownLatch / Semaphore

独占模式一次只放一个线程；共享模式可同时放多个。区别就在 tryAcquireShared 返回值 ≥ 0 时会向后传播，连续唤醒多个等待线程。

CountDownLatch(3) 的 state 之旅

CountDownLatch latch = new CountDownLatch(3);   // state = 3

// 工作线程：
latch.countDown();   // tryReleaseShared: CAS state 3 → 2
latch.countDown();   // CAS state 2 → 1
latch.countDown();   // CAS state 1 → 0  ← 归零! 触发 doReleaseShared 唤醒所有等待者

// 主线程：
latch.await();       // tryAcquireShared: state != 0 → 返回负数 → park 阻塞
                     // 直到 state == 0 → 返回正数 → 放行，且传播唤醒其他 await 线程
操作动了 state 吗做了什么
countDown()是，CAS 减 1归零时唤醒所有 await 线程（共享传播）
await()否，只读 statestate≠0 则入队 park；state==0 直接放行
state 语义复用的威力：ReentrantLock 把 state 当「重入计数」，CountDownLatch 当「倒计数」，Semaphore 当「许可数」。同一个 volatile int，三种锁，全靠子类的 tryAcquire/tryRelease 解释。这就是 AQS 被誉为「J.U.C 基石」的原因。

彩蛋：ThreadPoolExecutor.Worker 为什么是「不可重入」锁（接 L06）

线程池里的 Worker extends AbstractQueuedSynchronizer，它的 tryAcquire 写死只允许 0→1，不做重入累加：

protected boolean tryAcquire(int unused) {
    if (compareAndSetState(0, 1)) {       // 只能从 0 抢到 1
        setExclusiveOwnerThread(Thread.currentThread());
        return true;
    }
    return false;                          // 已锁 → 直接失败，不重入
}

为什么故意不可重入：Worker 的锁状态被用来表达「这个线程是否正在执行任务」。如果可重入，调用 setCorePoolSize 等方法时去 interruptIdleWorkers()，可能在任务执行中（已持锁）又获得锁，误判为「空闲」而中断正在干活的线程。不可重入 → 正在执行任务的 Worker 必然持锁 → tryLock() 失败 → 不会被当成 idle 中断。

6. CAS 基石与 ABA 问题

AQS 所有「无锁」操作都靠 CAS。CAS = Compare And Swap，一条 CPU 原子指令（x86 的 cmpxchg）：

// Unsafe 提供，比较内存值==expect 才写入 update，返回是否成功
public final native boolean compareAndSwapInt(
    Object o, long offset, int expect, int update);

为什么 state 必须 volatile：CAS 只保证「比较+交换」这一步原子，但线程 A 改了 state 后，线程 B 必须立即看到新值，否则 B 拿旧值去 CAS 会反复失败、甚至读到脏数据导致逻辑错误。volatile 提供可见性 + 禁止重排序，是 CAS 正确工作的前提。去掉 volatile，AQS 队列状态在多核间不可见 → 永久空转或丢失唤醒。

ABA 问题

线程1 读到 state = A，准备 CAS(A → C)
线程2 趁机：A → B → A   (改回 A 了)
线程1 CAS(A → C) 成功！但中间发生过变化，它毫不知情 = ABA

解法：加版本号。AtomicStampedReference 把「值 + stamp」一起 CAS，每次修改 stamp + 1，A→B→A 时 stamp 从 1→2→3，第三次的 A 已不是原来的 A。

面试话术（项目锚定）

Q：synchronized 和 ReentrantLock 怎么选？

「我的默认选择是 synchronized——JDK 6 以后有锁升级，无竞争时几乎零开销，且自动释放、不会忘记 unlock。只有需要这几个能力时才上 ReentrantLock：可中断等待（lockInterruptibly）、超时获取（tryLock 避免死等）、公平锁、多个条件队列（Condition 分离生产/消费等待）。比如我在 clickhouse-mcp-server 里给连接池做获取连接的逻辑，就用 tryLock(timeout) 来避免线程无限阻塞拖垮整个工具调用。」

Q：AQS 是什么，ReentrantLock 怎么基于它实现？

「AQS 是 J.U.C 的并发基础框架，核心就三件：一个 volatile int 的 state、一个 CAS 维护的 CLH 变体双向等待队列、以及一套 acquire/release 模板方法。它把『排队、阻塞、唤醒』这些复杂的同步骨架写死在父类，子类只需要重写 tryAcquire/tryRelease 来定义 state 的语义。ReentrantLock 就是把 state 解释为重入次数：0 表示空闲，CAS 抢到设为 1，同一线程再进来就直接累加，解锁减到 0 才唤醒队列里的后继线程。我在排查线程池源码时发现 Worker 也是继承 AQS 实现的不可重入锁,用来标记线程是否在执行任务。」

Q：本地锁和你用过的 Redisson 分布式锁有什么共性？

「设计上高度同构。三个维度对照：可重入——ReentrantLock 用 AQS 的 state 累加 + exclusiveOwnerThread 判断持有者，Redisson 用 Redis Hash 的 count + uuid:threadId；等待队列——AQS 在 JVM 堆里维护 CLH 双向链表 park/unpark，Redisson 靠 Redis pub/sub 订阅 channel 等待唤醒；差异在于分布式锁多了一个本地锁没有的问题——持有者跨进程崩溃，所以 Redisson 需要 TTL + WatchDog 续期，而本地锁的持有者和锁在同一个 JVM，线程死了锁状态也随之消失，不需要续期。」
