ConcurrentHashMap 源码深度

JDK7 分段锁 → JDK8 CAS + synchronized + Node 链表/红黑树

Lesson 16 · 2026-06-21 · ⏱ 40 min · 进阶

开篇连接

「ConcurrentHashMap 怎么保证线程安全？JDK7 和 JDK8 有什么区别？」是 Java 并发面试的开场必杀题，而它正是 L14（CAS / synchronized 锁升级 / volatile / Unsafe） 这些底层原语的「实战出口」——把零散的同步工具拼装成一个真实数据结构。学完这一课，你能在 clickhouse-mcp-server 里用它做 query 结果的并发安全本地缓存，并把它和 L15 的 Go sync.Map 放到同一张并发 map 设计权衡表里讲。

前置检查

前置 1（来自 L14）：CAS 的三个参数是什么？为什么 CAS 操作的目标变量通常要配 volatile？

查看答案

三参数：内存位置 V、预期旧值 A、新值 B——当且仅当 V 当前值 == A 才把它原子地改成 B。CAS 只保证「比较并交换」这一步原子，可见性靠 volatile：保证一个线程改了值其他线程立即看到，否则会基于脏值做 CAS。这正是 ConcurrentHashMap 里 table 必须 volatile、写桶用 Unsafe 的原因。

前置 2（来自 L14）：synchronized 锁升级四个阶段是什么？为什么 JDK8 敢用 synchronized 锁单个桶头而不怕慢？

查看答案

无锁 → 偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（OS 互斥量挂起）。低竞争时停在偏向/轻量级，几乎不进内核，开销很小。锁住的又只是一个桶头节点，竞争被哈希打散，所以 synchronized 在这里反而比为每个 Segment 持有一个 ReentrantLock 更轻。

前置 3：HashMap 链表转红黑树的两个条件是什么？HashMap 在多线程下分别有什么问题（JDK7 / JDK8）？

查看答案

转树双条件：链表长度 ≥ 8 且 table 容量 ≥ 64（容量不足时先扩容而非转树）。多线程问题：JDK7 头插法在并发扩容时形成环形链表 → get 死循环 100% CPU；JDK8 改尾插消除了死循环，但并发 put 仍会丢数据 / 覆盖，且仍非线程安全。

1. JDK7：分段锁（Segment + ReentrantLock）

JDK7 的核心思路是「把一把大锁拆成 N 把小锁」。整个 map 是一个 Segment[] 数组，每个 Segment 本身继承自 ReentrantLock，内部维护一个小型的 HashEntry[]（相当于一个迷你 HashMap）。

ConcurrentHashMap
  └─ Segment[]  // 默认 16 个，初始化后不可变 → 并发度固定 16
      └─ Segment extends ReentrantLock
          └─ HashEntry[]  // 数组 + 链表

定位元素需要两次哈希：高位 hash 选 Segment，低位 hash 选 HashEntry 桶。put 时只锁住命中的那一个 Segment，其他 Segment 可并发写。

// JDK7 put 简化：先定位 Segment，再在段内加锁
final V put(K key, int hash, V value, boolean onlyIfAbsent) {
    HashEntry<K,V> node = tryLock() ? null    // 1. 先 tryLock 抢这个 Segment 的锁
        : scanAndLockForPut(key, hash, value); //    抢不到则自旋等待，顺便预热遍历
    V oldValue;
    try {
        HashEntry<K,V>[] tab = table;
        int index = (tab.length - 1) & hash;  // 2. 段内第二次哈希定位桶
        // ... 遍历链表，命中则覆盖，否则头插新节点 ...
    } finally {
        unlock();                              // 3. 释放该 Segment 的 ReentrantLock
    }
    return oldValue;
}

JDK7 的硬伤：① 并发度被 Segment 数量写死（默认 16），再多线程也只能 16 路并行；② 每个 Segment 常驻一个 ReentrantLock 对象，内存开销固定；③ 两次哈希定位，寻址链路长；④ size() 要尝试锁所有段。

即时检验：JDK7 中如果你 new 时不指定 concurrencyLevel，两个线程操作落在同一个 Segment 的不同桶上，它们能并发写吗？

查看答案

不能。锁的粒度是「整个 Segment」，不是单个桶。只要落在同一个 Segment，即使桶不同也会互相阻塞。这正是 JDK8 要把锁粒度细化到单桶的动机。

2. JDK8 重构：抛弃 Segment，CAS + synchronized 锁单桶

JDK8 彻底废弃了 Segment 分段锁，结构回归到和 HashMap 几乎一致的 Node[] table（数组 + 链表 + 红黑树），但把同步策略换成了「CAS 写空桶 + synchronized 锁桶头」。锁粒度从「一段」细化到「一个桶」，并发度 = 桶的数量（随扩容增长），不再被固定。

// JDK8 关键字段
transient volatile Node<K,V>[] table;        // 主表，volatile 保证引用可见
private transient volatile Node<K,V>[] nextTable; // 扩容时的新表
private transient volatile long baseCount;     // 基础计数（低竞争）
private transient volatile int sizeCtl;         // 控制位：负数=初始化/扩容中

static class Node<K,V> {
    final int hash;
    final K key;
    volatile V val;            // val 和 next 都是 volatile → get 无锁可见
    volatile Node<K,V> next;
}

volatile 陷阱（高频考点）：table 字段是 volatile，但这只保证数组引用本身可见，不保证数组元素 table[i] 可见！Java 没有「volatile 数组元素」语法。所以读写桶必须用 Unsafe 的内存屏障版本：

// 三个核心 Unsafe 操作 —— 不能直接写 table[i]
static final <K,V> Node<K,V> tabAt(Node<K,V>[] tab, int i) {
    return (Node<K,V>) U.getReferenceAcquire(tab, ...); // volatile 语义读元素
}
static final <K,V> boolean casTabAt(Node<K,V>[] tab, int i,
                                       Node<K,V> c, Node<K,V> v) {
    return U.compareAndSetReference(tab, ..., c, v);  // CAS 写空桶
}
static final <K,V> void setTabAt(Node<K,V>[] tab, int i, Node<K,V> v) {
    U.putReferenceRelease(tab, ..., v);              // volatile 语义写元素
}

即时检验：既然 table 已经是 volatile，为什么读一个桶还要用 tabAt() 而不是直接 table[i]？

查看答案

因为 volatile 修饰的是「数组引用变量」，不是数组里每个元素的槽位。直接 table[i] 是普通读，可能读到其他线程刚 CAS 写入但尚未对本线程可见的旧值。tabAt() 通过 Unsafe 的 acquire 语义读，确保拿到最新元素。

3. putVal 完整流程（源码注释级别）

这是面试最爱让你「手撕流程」的地方。JDK8 的 putVal 是一个 for(;;) 自旋，根据当前桶的状态分四条路径：

final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException(); // ① 不允许 null
    int hash = spread(key.hashCode());     // 扰动：高 16 位异或低 16 位，且抹掉符号位
    int binCount = 0;
    for (Node<K,V>[] tab = table;;) {       // 自旋直到成功
        Node<K,V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();             // ② 表未初始化 → 用 CAS+sizeCtl 抢初始化权
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null,        // ③ 空桶 → CAS 无锁写头节点
                        new Node<K,V>(hash, key, value)))
                break;                      //    CAS 成功直接退出；失败则自旋重试
        }
        else if ((fh = f.hash) == MOVED)    // ④ 桶头是 ForwardingNode(hash=-1)
            tab = helpTransfer(tab, f);    //    说明正在扩容 → 来帮忙搬迁
        else {
            V oldVal = null;
            synchronized (f) {              // ⑤ 非空桶 → 只锁这个桶头节点 f
                if (tabAt(tab, i) == f) {   //    double-check 头没被换走
                    if (fh >= 0) {           //    fh>=0 是链表
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            // 命中 key → 覆盖 val；到尾部 → 尾插新节点
                        }
                    } else if (f instanceof TreeBin) {
                        // 红黑树 → 走 putTreeVal
                    }
                }
            }
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)  // ⑥ 链表 ≥8 → treeifyBin
                    treeifyBin(tab, i);            //    但容量 <64 时它会改为扩容
                break;
            }
        }
    }
    addCount(1L, binCount);   // ⑦ 计数 +1，并按需触发扩容
    return null;
}

注意 treeifyBin 内部的双条件守卫：

private final void treeifyBin(Node<K,V>[] tab, int index) {
    if (tab.length < MIN_TREEIFY_CAPACITY)  // MIN_TREEIFY_CAPACITY = 64
        tryPresize(tab.length << 1);        // 容量不足 → 先扩容，不转树
    else if (...) { /* synchronized 锁桶头，链表转 TreeBin */ }
}

一句话记忆 putVal：空桶 CAS、撞桶 synchronized、遇 ForwardingNode 帮搬家、链长八且容六四才转树、最后 addCount。

即时检验：三种情形分别走哪条路径？(a) 目标桶为空；(b) 目标桶已有链表且未在扩容；(c) 目标桶头节点 hash == -1。

查看答案

(a) casTabAt 无锁 CAS 写头节点；(b) synchronized(f) 锁住桶头遍历链表覆盖/尾插；(c) hash=-1 即 MOVED，是 ForwardingNode，走 helpTransfer 协助扩容后再重试。

4. get 为什么不需要加锁

get 全程无锁，靠的是 Node.val 和 Node.next 都是 volatile，加上 tabAt 的 volatile 读语义。读到的要么是旧值要么是新值，但绝不会读到「写了一半」的中间态。

public V get(Object key) {
    Node<K,V>[] tab; Node<K,V> e, p; int n, eh; K ek;
    int h = spread(key.hashCode());
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (e = tabAt(tab, (n - 1) & h)) != null) {   // volatile 读桶头
        if ((eh = e.hash) == h) { /* 头节点命中 */ }
        else if (eh < 0)              // hash<0：ForwardingNode 或 TreeBin → 走 e.find()
            return (p = e.find(h, key)) != null ? p.val : null;
        while ((e = e.next) != null) { // 遍历链表，next 是 volatile
            if (e.hash == h && keyEquals(...)) return e.val;
        }
    }
    return null;
}

正在扩容时 get 也安全：如果桶头是 ForwardingNode（hash<0），e.find() 会被转发到 nextTable 上去查，读操作永远能找到正确的数据。这也解释了为什么 ConcurrentHashMap 的迭代器是弱一致性的——它反映的是创建后某一时刻的状态，不抛 ConcurrentModificationException。

5. size() 与 CounterCell：分散热点的计数

如果用一个 volatile long count 做全局计数，每次 put 都要 CAS 它 → 高并发下这个单点变成热点，CAS 频繁失败自旋。JDK8 借用了 LongAdder 的同源思想：把计数拆散到多个 cell 上，最后求和。

private transient volatile long baseCount;        // 无竞争时直接 CAS 这个
private transient volatile CounterCell[] counterCells; // 竞争时分散到数组

private final void addCount(long x, int check) {
    CounterCell[] cs; long b, s;
    if ((cs = counterCells) != null ||
        !U.compareAndSetLong(this, BASECOUNT, b = baseCount, s = b + x)) {
        // CAS baseCount 失败（有竞争）→ 落到某个 CounterCell 上 CAS
        // 通过线程的 probe 哈希选 cell，把冲突打散到不同缓存行
        fullAddCount(x, ...);
    }
    // ... 检查是否需要扩容 ...
}

public int size() {
    long n = sumCount();   // baseCount + 所有 counterCells[i].value 求和
    return n < 0 ? 0 : (n > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) n);
}

size() 是估算值：sumCount() 求和时不加锁，求和过程中其他线程仍在 put/remove，所以返回的是「某一瞬间的近似值」，弱一致。这是「用一致性换吞吐」的典型设计——和 LongAdder 同源。

即时检验：为什么不用一个简单的 AtomicLong 做计数，而要搞 baseCount + CounterCell[]？

查看答案

AtomicLong 是单点 CAS，高并发下所有线程争同一个变量，CAS 失败率高、自旋浪费 CPU、还有缓存行抖动。CounterCell[] 把计数分散到多个槽位（不同缓存行），各线程按 probe 哈希落到不同 cell，把热点打散，读时求和——这就是 LongAdder 的核心思想，用空间和读取时的一致性换写入吞吐。

6. 扩容协作：sizeCtl、ForwardingNode、helpTransfer

JDK8 扩容最精妙的地方是「多线程协作迁移」——扩容时其他线程的 put 不会干等，而是会主动来帮忙搬运。

sizeCtl 的值含义
正数下次扩容的阈值（容量 × 0.75）
-1表正在初始化
负数（非 -1）正在扩容，高 16 位是扩容戳，低 16 位是「参与扩容的线程数 + 1」

迁移按桶分段进行：每个线程通过 CAS 领取一段桶区间（stride，默认最小 16）去搬。一个桶搬完后，在旧表的该位置放一个 ForwardingNode（hash = MOVED = -1） 作为占位标记。

线程A 触发扩容 → 建 nextTable，sizeCtl 变负
线程B put 时撞到 ForwardingNode → helpTransfer(tab, f) 加入搬迁
某桶搬完 → 旧表该位置 setTabAt 一个 ForwardingNode
读到 ForwardingNode 的 get → 自动转发到 nextTable 查找
全部搬完 → table = nextTable，sizeCtl 回到正阈值

这是 L14 的 CAS + volatile state 在数据结构层的极致应用：sizeCtl 就像 AQS 的 state，多线程靠 CAS 它来协调「谁初始化、谁扩容、扩容到什么程度」，没有一把全局大锁。

经典陷阱：复合操作不原子。下面这段在 ConcurrentHashMap 上仍然有竞态——单个 get/put 各自线程安全，但「先判断再写入」之间可能被插入：

// ❌ 错误：check-then-act 不是原子的
if (!map.containsKey(key)) {   // 线程 A、B 可能同时通过这里
    map.put(key, compute());   // 两个都 put，后者覆盖前者
}

// ✅ 正确：用原子的 putIfAbsent / computeIfAbsent
map.putIfAbsent(key, value);
map.computeIfAbsent(key, k -> expensiveCompute(k)); // 还保证 compute 只跑一次

面试话术

Q：「ConcurrentHashMap 怎么保证线程安全？JDK7 和 JDK8 区别？」

我会分三层讲。第一层结构：JDK7 是分段锁，外层 Segment 数组，每个 Segment 继承 ReentrantLock，锁粒度是一整段，并发度固定为 Segment 数（默认 16）；JDK8 抛弃了 Segment，回到 Node 数组加链表/红黑树，锁粒度细化到单个桶头。第二层写策略：JDK8 的 putVal 分三种情况——空桶用 casTabAt 无锁 CAS 写入，撞桶用 synchronized 只锁这一个头节点，遇到 hash 为 -1 的 ForwardingNode 说明正在扩容、就调 helpTransfer 来协助搬迁。第三层读：get 全程无锁，因为 Node 的 val 和 next 都是 volatile，配合 tabAt 的 volatile 读语义，永远读不到中间态。在我做的 clickhouse-mcp-server 里，我就用它做 query 结果的本地缓存，读远多于写，无锁读这点收益很直接。

Q：「为什么 JDK8 不继续用 ReentrantLock，而改用 synchronized？」

关键在 synchronized 在 JDK6 之后有了锁升级优化：偏向锁、轻量级锁、重量级锁逐级升级，低竞争时停在偏向或轻量级阶段、靠 CAS 自旋根本不进内核，开销很小。而锁住的又只是一个桶头节点，竞争已经被哈希打散了，绝大多数桶根本不会膨胀到重量级。相比之下 JDK7 给每个 Segment 常驻一个 ReentrantLock 对象，内存开销固定且偏重。所以在「锁粒度足够细 + 锁升级足够轻」的前提下，synchronized 反而更省内存、更轻量。这跟 L14 讲的锁升级是同一套底层逻辑。

Q：「size() 准确吗？怎么统计的？」

不绝对准确，是一个弱一致的估算值。它没有用单个 AtomicLong，因为高并发下单点 CAS 会成为热点、失败自旋、缓存行抖动。JDK8 借鉴 LongAdder 的思想，用 baseCount 加上一个 CounterCell 数组：无竞争时直接 CAS baseCount，一旦冲突就按线程的 probe 哈希散落到不同的 CounterCell 上，把热点打散到不同缓存行；size() 时把 baseCount 和所有 cell 求和。求和过程不加锁，所以拿到的是某一瞬间的近似值——这是典型的用一致性换吞吐。

Q2. JDK8 putVal 时，目标桶头节点的 hash == -1（MOVED），应该走哪条路径？

A. casTabAt 无锁写入 B. synchronized 锁住头节点遍历链表 C. helpTransfer 协助扩容后重试 D. 抛 ConcurrentModificationException

Q3. 为什么 size() 用 baseCount + CounterCell[] 而不用单个 AtomicLong？

A. AtomicLong 不支持 long 类型 B. 单点 CAS 在高并发下成热点、失败自旋、缓存行抖动；分散到多 cell 可打散热点（LongAdder 思想） C. CounterCell 能让 size() 变得绝对精确 D. 为了节省内存

Q4. table 已是 volatile，为什么读桶还要用 tabAt()？

A. tabAt 比 table[i] 快 B. volatile 只保证数组引用可见，不保证元素可见；直接 table[i] 可能读到旧值 C. table[i] 会抛 ArrayIndexOutOfBounds D. tabAt 会自动加锁

Q5. 一个桶链表长度达到 8，但 table 容量只有 32，会发生什么？

A. 立即把链表转成红黑树 B. 不转树，因为容量 < 64，treeifyBin 会改为先扩容 C. 抛异常 D. 把链表清空

Q6. 在 ConcurrentHashMap 上做「if (!containsKey) put」会有什么问题？

A. 没问题，containsKey 和 put 都是线程安全的 B. check-then-act 复合操作不原子，两线程可能都通过判断后都 put，应改用 putIfAbsent/computeIfAbsent C. 会死锁 D. containsKey 会抛 NPE

Q7. ConcurrentHashMap（锁分桶）与 Go sync.Map 的核心设计差异是？

A. 两者完全相同 B. CHM 靠锁分桶（写锁单桶、读无锁）；sync.Map 靠 read/dirty 双 map 读写分离，最适合读多写少 C. sync.Map 用 ReentrantLock D. CHM 不支持并发读
