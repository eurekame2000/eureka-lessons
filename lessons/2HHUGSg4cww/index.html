Agent 可观测性

Span 为单位的推理追踪 · OpenTelemetry GenAI 语义约定

Lesson 10 · 2026-06-20 · ⏱ 25 min

面试情境：你正在面高级后端/AI Agent 岗位，面试官问："你的 Agent 在生产上是怎么保证可观测性的？如果 Agent 回答错了，你怎么知道？怎么定位？"

1. 传统盲区：为什么 "up" 不够

传统 APM 告诉你三件事：服务通不通、响应慢不慢、有没有 500。对 Agent 来说这些远远不够。

Agent 的失败模式是 静默语义错误：

  • Agent 返回了一个格式正确的回复，但决策错了（选了错误的工具）
  • Agent 陷入推理循环，token 消耗暴增，但最后一个 span 依然是 "success"
  • 子 Agent 返回了有毒的结果，父 Agent 没有检查直接用它做了下一步

这些错误不会产生 HTTP 5xx。唯一的观测手段是 追踪 Agent 的每个推理步骤——即 span-per-tick tracing：每一轮推理（reasoning tick）生成一个独立的 span，嵌套在完整请求的 trace 之下。

核心洞察 ：Agent 可观测性 = 追踪 意图和执行路径，而不仅仅是输入/输出。你需要回答的不只是 "Agent 回复了什么"，而是 "Agent 为什么选择了这个工具？它考虑了哪些选项？花了多少 token？"

2. 四类 Span（核心知识）

OpenTelemetry GenAI SIG 定义的 gen_ai.* 语义约定将 Agent 追踪拆为四类 span，覆盖所有失败面：

Span 类型OTel span name捕获的失败
invoke_agent invoke_agent {agent.name} Agent 调度失败、上下文超长、子 Agent 未返回
chat chat {gen_ai.request.model} LLM 调用失败、token 突发、finish_reason 异常
execute_tool execute_tool {gen_ai.tool.name} 工具抛异常、超时、返回非法数据、重复调用
memory / state memory.{operation} 上下文丢失、summarization 漂移、状态污染

这四个 span 类型构成一棵 层次化 span 树，一个 trace 对应一个用户请求的全部执行路径：

Trace 0x1a2b — 用户请求 "帮我查一下昨天的销售数据"
└── [invoke_agent] sales-agent
    ├── [chat] gpt-4o (system prompt + 用户问题)
    │   └── finish_reason: tool_calls
    ├── [execute_tool] query_clickhouse
    │   ├── gen_ai.tool.call.arguments: {"sql":"SELECT ..."}
    │   └── gen_ai.tool.call.result: {"rows": 42, ...}
    ├── [memory.write] context_update (工具结果注入)
    └── [chat] gpt-4o (工具结果 → 自然语言回复)
        └── finish_reason: stop

每一个 span 都记录了：输入、输出、时间戳、错误状态、parent span ID。有了这棵树，你可以回答任何问题——"第几步花了最久？哪一步抛了异常？这个 tool call 的参数是什么？"

3. OTel GenAI 语义约定速览

OpenTelemetry GenAI SIG 从 2024 年 4 月开始制定标准化属性名。截至 2026 年 5 月（v1.41），核心约定如下：

属性作用用在
gen_ai.request.model模型名称（如 gpt-4o, gemini-2.0-flash）chat span
gen_ai.usage.input_tokens输入 token 数chat span
gen_ai.usage.output_tokens输出 token 数chat span
gen_ai.response.finish_reasonsstop / tool_calls / length / content_filterchat span
gen_ai.tool.name工具名称execute_tool span
gen_ai.tool.call.arguments工具调用参数（JSON）execute_tool span
gen_ai.tool.call.result工具返回结果execute_tool span
gen_ai.agent.nameAgent 名称/IDinvoke_agent span
注意 ：GenAI 语义约定目前仍是 Development 状态，属性名可能变化。使用 OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental 明确锁定版本。

内容采集模式

默认情况下，OTel GenAI 约定 不采集 prompt/completion 内容——只采集元数据（模型名、token 数、时长）。内容采集有三种模式：

  1. 元数据只（默认）——无敏感数据风险，但无法 debug 质量
  2. 内容内联（opt-in）——将 messages 作为 span 属性写入，适合 dev 环境
  3. 内容外置（推荐生产）——内容存外部存储，span 上只挂引用

4. 动手练习：用 Java OTel SDK 追踪一个 Tool Call

下面代码展示如何手工创建 execute_tool span 并附加 gen_ai.* 属性。实际场景中你会在工具的 wrapper 或 interceptor 中做这件事。

// 依赖：io.opentelemetry:opentelemetry-api:1.44.0
import io.opentelemetry.api.GlobalOpenTelemetry;
import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.Tracer;
import io.opentelemetry.context.Scope;
import io.opentelemetry.api.common.Attributes;
import io.opentelemetry.api.common.AttributeKey;

public class AgentTracer {

    static final Tracer tracer = GlobalOpenTelemetry.getTracer("agent-tracer");

    public static  T traceToolCall(
        String toolName,      // 工具名
        String arguments,     // JSON 参数
        ThrowingSupplier<T> fn  // 真正的工具逻辑
    ) throws Throwable {
        // 创建 execute_tool 子 span
        Span span = tracer.spanBuilder("execute_tool " + toolName)
            .setSpanKind(SpanKind.INTERNAL)
            .setAttribute("gen_ai.operation.name", "execute_tool")
            .setAttribute("gen_ai.tool.name", toolName)
            .setAttribute("gen_ai.tool.call.arguments", arguments)
            .start();

        try (Scope ignored = span.makeCurrent()) {
            T result = fn.get();
            String resultStr = result == null ? "" : result.toString();

            // 截断大结果，避免 span 超尺寸
            if (resultStr.length() > 10000) {
                resultStr = resultStr.substring(0, 10000) + "... [truncated]";
            }
            span.setAttribute("gen_ai.tool.call.result", resultStr);
            span.setStatus(StatusCode.OK);
            return result;
        } catch (Throwable t) {
            span.recordException(t);
            span.setStatus(StatusCode.ERROR, t.getMessage());
            throw t;
        } finally {
            span.end();
        }
    }

    @FunctionalInterface
    public interface ThrowingSupplier<T> {
        T get() throws Throwable;
    }
}

用法，以一个 ClickHouse 查询工具为例：

// 在 Agent 的 Tool Router 中
String result = AgentTracer.traceToolCall(
    "query_clickhouse",
    "{\"sql\": \"SELECT count(*) FROM events WHERE date = '2026-06-19'\"}",
    () -> clickHouseClient.query("SELECT count(*) FROM events WHERE date = '2026-06-19'")
);

// 使用 Tracer 手动构建 span 树
Span rootSpan = tracer.spanBuilder("invoke_agent sales-query")
    .setSpanKind(SpanKind.INTERNAL)
    .setAttribute("gen_ai.operation.name", "invoke_agent")
    .setAttribute("gen_ai.agent.name", "sales-query")
    .start();

try (Scope scope = rootSpan.makeCurrent()) {

    // Step 1: LLM 调用
    Span llmSpan = tracer.spanBuilder("chat gpt-4o")
        .setSpanKind(SpanKind.CLIENT)
        .setAttribute("gen_ai.operation.name", "chat")
        .setAttribute("gen_ai.request.model", "gpt-4o")
        .setAttribute("gen_ai.usage.input_tokens", 450)
        .setAttribute("gen_ai.usage.output_tokens", 120)
        .setAttribute("gen_ai.response.finish_reasons", "tool_calls")
        .start();
    llmSpan.end();

    // Step 2: 工具调用
    Span toolSpan = tracer.spanBuilder("execute_tool query_clickhouse")
        .setSpanKind(SpanKind.INTERNAL)
        .setAttribute("gen_ai.operation.name", "execute_tool")
        .setAttribute("gen_ai.tool.name", "query_clickhouse")
        .setAttribute("gen_ai.tool.call.arguments",
            "{\"sql\": \"SELECT sum(revenue) FROM sales WHERE date = '2026-06-19'\"}")
        .start();

    // 模拟调用
    String result = clickHouseClient.query("SELECT sum(revenue) FROM sales ...");
    toolSpan.setAttribute("gen_ai.tool.call.result", result);
    toolSpan.setStatus(StatusCode.OK);
    toolSpan.end();

} catch (Exception e) {
    rootSpan.recordException(e);
    rootSpan.setStatus(StatusCode.ERROR, e.getMessage());
    throw e;
} finally {
    rootSpan.end();
}

📝 参考答案

2. 三个工具各一个 execute_tool 子 span，失败的设 StatusCode.ERROR 并 recordException()。即使异常被 catch，span 应该如实记录——"已发生错误，被业务逻辑 catch 了"是一种信息，不是掩饰的理由。

5. 面试应答结构

如果面试官问"你的 Agent 怎么监控"，按这个结构回答：

  1. 分层："Agent 可观测性和传统不一样。传统 APM 看请求量/错误率/延迟就够了，Agent 的失败是静默语义错误，必须追踪推理路径。"
  2. Span 树："我们用 OTel GenAI 语义约定，每个用户请求是一条 trace，里面包含 invoke_agent、chat、execute_tool、memory 四种 span。每个 span 记录输入输出、token 消耗、耗时。"
  3. 评估层："trace 之上还要跑 LLM-as-Judge 评估，对采样 trace 打分，判断回复质量是否达标。"
  4. 成本："采样策略：100% 元数据 + 10-20% 内容 + 100% 错误 trace 全量。内容外置存储，PII 在 Collector 层脱敏。"

6. Quiz

Q1: Agent 失败为什么传统 APM 看不到？

✅ 答案

C — Agent 失败通常不出 crash 或 500，而是输出质量下降、选错工具、陷入循环。这些不会被传统 apdex 或 error rate 捕获。

Q2: 以下哪个不是 OTel GenAI 定义的 Agent span 类型？

✅ 答案

C — 四个核心类型是 invoke_agent、chat、execute_tool、memory/state。

Q3: 生产中推荐的内容采集策略？

✅ 答案

A — 元数据（结构 span）100% 保证成本和延迟统计完整；内容采样 10-20% 够做 LLM eval；错误 trace 全量确保可 debug。PII 匹配的 trace 内容 0% 采集。
