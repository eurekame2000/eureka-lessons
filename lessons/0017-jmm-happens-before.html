开篇连接

前两课你已经在用 volatile：L14 里 AQS 的 state 是 volatile，L16 里 ConcurrentHashMap 的 table 是 volatile——但当时我们只说"它保证可见性"，没讲为什么、靠什么落地。这一课就是补上那块理论底座：JMM 与 happens-before。学完你能在高级面试里把"volatile 能保证原子性吗 / happens-before 有哪些规则 / DCL 为什么要 volatile"从背诵升级为现场推导，并在 clickhouse-mcp-server 多线程处理请求时，用 happens-before 论证哪些状态字段需要 volatile、哪些靠 synchronized 边界安全发布。

前置检查

前置 1（来自 L14 / L16）：volatile 此前在哪两节课出现过、分别修饰了什么字段、起什么作用？

查看答案

L14：AQS 的同步状态 state 是 volatile int——多个线程通过 CAS 抢锁，必须保证一个线程改了 state 其他线程立刻看到，否则会基于脏值 CAS。L16：ConcurrentHashMap 的 table 引用是 volatile，且桶元素读写走 tabAt（Unsafe 的 volatile 语义读），保证 get 无锁也能看到最新桶头与扩容标记。两处的共同诉求都是可见性——而可见性的理论根据，就是本课要讲的 happens-before。

前置 2（来自 L14）：CAS 三个参数是什么？为什么 CAS 的目标变量通常要配 volatile？

查看答案

三参数：内存位置 V、预期旧值 A、新值 B，当且仅当 V==A 时原子地写入 B。CAS 只保证"比较并交换"这一步原子，不保证可见性；volatile 负责让别的线程看到最新值，避免基于过期值做 CAS。本课会用 happens-before 严格解释这件事。

前置 3：并发的三大问题是原子性 / 可见性 / 有序性，各指什么？本课主攻哪两个？

查看答案

原子性：一组操作不可分割（i++ 其实是读-改-写三步，非原子）。可见性：一个线程的写，另一个线程能否、何时看到。有序性：代码书写顺序与实际执行顺序是否一致（重排序会打乱）。volatile 主攻可见性 + 有序性，不解决原子性——这是本课的核心张力。

1. 为什么需要 JMM：硬件根源

JMM（Java Memory Model）不是"JVM 内存结构"（堆/栈/方法区那套），而是一套抽象规范：它定义"一个线程的写，在什么条件下对另一个线程可见"。为什么需要这层抽象？因为现代 CPU 为了性能，撒了三个谎。

谎言一：缓存让"主存"形同虚设（可见性问题）

CPU 不会每次都直连主存——中间隔着 L1/L2/L3 多级缓存，还有store buffer（写缓冲）。线程 A 在 Core 0 上把 flag = true 写进自己的 store buffer，还没刷回主存；线程 B 在 Core 1 上读到的还是缓存里的旧 false。缓存一致性协议（MESI）最终会同步，但"最终"可能是几十纳秒后，甚至在某些等待条件下"永远不主动刷"。

经典死循环 bug：

boolean running = true;            // 没加 volatile
// 线程 A（工作线程）
while (running) { doWork(); }      // 可能永远跑下去
// 线程 B（主线程）
running = false;                   // A 可能永远看不到

JIT 甚至会把 running 提到寄存器里，while 变成 while(true)。这就是 batch-import-optimizer 里"停止标志位"的经典场景——必须 volatile boolean running。

谎言二：指令乱序执行（有序性问题）

CPU 流水线、编译器优化会重排指令；只要在单线程内看结果一致（as-if-serial）就允许重排。单线程无感，但多线程下，另一个线程会观察到"半成品"的中间状态。

谎言三：复合操作不是原子的（原子性问题）

i++ = load i → i+1 → store i，三步之间可能被打断。这个问题 volatile 解决不了，要靠锁或 CAS（见第 5 节）。

一句话总结：JMM 是程序员和 JVM/CPU 之间的"契约"——你按 happens-before 规则写代码，JVM 保证你看到的结果符合直觉；你不按规则写，JVM 就放开手脚优化，出现可见性/有序性 bug 不怪它。

检验 1. 上面 running 死循环的根本原因是？

A. running 是 boolean，类型太小放不下 B. 工作线程把 running 缓存在本地（缓存/寄存器），看不到主线程的写——可见性问题 C. running 没初始化 D. while 循环本身有 bug

2. 指令重排的三个层级 & as-if-serial

"重排序"不是一个东西，而是三层叠加：

层级谁干的例子
编译器重排javac / JIT调整无依赖语句顺序、把变量提到寄存器
CPU 乱序执行处理器流水线指令级并行，谁的操作数先就绪先执行
内存系统重排store buffer / 失效队列写先进 store buffer，对别的 core "延迟可见"，造成 StoreLoad 乱序

as-if-serial（似乎串行）语义：不管怎么重排，单线程程序的执行结果不能被改变。有数据依赖的操作不会重排（a=1; b=a+1; 不会换序），没依赖的才可能。

关键：as-if-serial 只在单线程内成立，多线程下"失效"。 因为它只保证本线程看自己的结果一致，不保证别的线程看到的顺序。经典示例：

// 初始 a=0, b=0；线程1 执行左侧，线程2 执行右侧
线程1: a = 1;   x = b;
线程2: b = 1;   y = a;
// 直觉上 x、y 不可能同时为 0
// 但 store buffer 重排后，x==0 && y==0 真的会发生

那程序员怎么在需要的时候关掉重排、建立跨线程的顺序保证？答案不是"逐条禁止重排"——而是用 JMM 给的高层规则：happens-before。

检验 2. as-if-serial 语义保证的是？

A. 多线程下所有操作顺序一致 B. 单线程内重排不改变本线程执行结果，但不约束跨线程可见顺序 C. 禁止一切重排序 D. 只对 volatile 变量生效

3. happens-before：JMM 的核心契约

happens-before（先行发生，记作 hb）不是"时间上先发生"，而是一个可见性 + 有序性的保证：如果操作 A hb 操作 B，那么 A 的结果对 B 可见，且 A 在 B 之前（在 B 看来）发生。注意——只要结果符合 hb 约束，JVM 仍可自由重排没被约束的部分。

八条规则（重点掌握加粗的 4 条）

#规则含义
1程序顺序规则同一线程内，前面的操作 hb 后面的操作（仅本线程视角）
2监视器锁规则一个锁的 unlock hb 后续对同一个锁的 lock
3volatile 规则对 volatile 变量的写 hb 后续对同一变量的读
4传递性A hb B 且 B hb C，则 A hb C（推导链的关键）
5线程启动规则Thread.start() hb 该线程内的任何操作
6线程终止规则线程内所有操作 hb 其他线程从该线程 join() 返回
7线程中断规则interrupt() hb 被中断线程检测到中断
8对象终结规则构造函数结束 hb finalize() 开始

会用规则做推导，比背规则重要 10 倍。 看这个例子怎么靠传递性建立可见性：

int data = 0;
volatile boolean ready = false;

// 线程 A（生产者）
data = 42;          // (1)
ready = true;       // (2) volatile 写

// 线程 B（消费者）
if (ready) {        // (3) volatile 读
    use(data);      // (4) 一定看到 data==42
}

推导链：(1) hb (2)（程序顺序）→ (2) hb (3)（volatile 写 hb 读）→ (3) hb (4)（程序顺序）→ 由传递性 (1) hb (4)，所以 (4) 读到的 data 必为 42。这就是 volatile 的"搭便车"效应：它不仅让自己可见，还把它之前的普通写一并发布出去。

没有 hb = 数据竞争（data race）。两个线程访问同一变量、至少一个是写、且它们之间没有 hb 关系 → 这就是数据竞争，结果未定义。判断并发 bug 的正确姿势：不是"凭感觉加 volatile"，而是检查关键读写之间能不能连出一条 hb 链。

检验 3. 上例若把 ready 的 volatile 去掉，会怎样？

A. 完全一样，没区别 B. 断了 (2)hb(3) 这一环，传递链失效——线程B可能读到 ready==true 但 data 还是 0 C. 编译报错 D. data 会变成原子的

4. volatile 底层：四种内存屏障

volatile 的两重语义（可见性 + 禁止重排）在底层靠内存屏障（memory barrier / fence）落地。JMM 规定 JIT 在 volatile 读写前后插入屏障：

屏障作用插入位置
StoreStore屏障前的普通写，先于 volatile 写刷出volatile 写之前
StoreLoad最强屏障：volatile 写 必须先于后续读完成（防写-读乱序）volatile 写之后
LoadLoadvolatile 读 先于后续普通读volatile 读之后
LoadStorevolatile 读 先于后续普通写volatile 读之后
... 普通写 ...
┌─ StoreStore 屏障
volatile 写
└─ StoreLoad 屏障 ← 最重，刷 store buffer 到主存
 
volatile 读
┌─ LoadLoad 屏障
└─ LoadStore 屏障
... 后续读/写 ...

这些抽象屏障最终映射到具体 CPU 的 fence 指令。x86 是强内存模型（TSO），只可能发生 StoreLoad 乱序，所以 volatile 写在 x86 上通常编译成一条 lock addl $0,(%rsp)（或 mfence），用 lock 前缀锁总线/缓存行 + 刷 store buffer + 让别的 core 缓存失效，同时充当全屏障；而读几乎零开销。ARM 是弱内存模型，需要更多显式 dmb 屏障。

可见性怎么落地：volatile 写 → 立刻把值刷回主存（store buffer flush）；volatile 读 → 让本地缓存中该变量失效，强制从主存重新读。有序性怎么落地：屏障禁止编译器和 CPU 把屏障两侧的指令跨越屏障重排。

检验 4. 四种屏障里最"重"、用来禁止 volatile 写与后续读乱序的是？

A. LoadLoad B. StoreStore C. StoreLoad（插在 volatile 写之后，开销最大） D. LoadStore

5. volatile 保证什么、不保证什么

volatile说明
可见性✅ 保证写刷主存、读失效本地缓存
有序性✅ 保证内存屏障禁止重排
原子性❌ 不保证仅保证单次读/单次写原子；复合操作不原子

面试必杀：volatile 能保证 i++ 原子吗？不能。 i++ 是 read-modify-write 三步：volatile 只保证每一步的读和写各自可见，但两个线程可以同时读到 5，各自 +1，都写回 6，丢了一次自增。

volatile int count = 0;
count++;   // 仍然有竞态！两线程跑 10000 次，结果常 < 20000

// 正确替代（回贴 L14 的 CAS）：
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();   // 内部 CAS 自旋，原子
// 或用锁 / synchronized 包住复合操作

记住口诀：volatile 适合"一写多读的状态标志位"，不适合"多线程累加/复合更新"。停止标志 running 用 volatile 完美；计数器要用 AtomicInteger 或 LongAdder。

检验 5. 下面哪个场景适合只用 volatile（不需要额外锁/原子类）？

A. 多线程对同一计数器做 count++ B. 一个线程设置 volatile boolean shutdown=true，多个工作线程读它决定是否退出 C. 多线程对同一余额做 balance -= amount D. 多线程 list.add()

6. DCL 双重检查锁：volatile 的杀手级用例

单例懒加载，想"加锁只在第一次、之后无锁"——这就是双重检查锁（Double-Checked Locking）。但不加 volatile 它是错的，原因正是重排序。

public class Singleton {
    private static volatile Singleton instance;   // ← 这个 volatile 必须有

    public static Singleton getInstance() {
        if (instance == null) {                  // 第一次检查（无锁，快路径）
            synchronized (Singleton.class) {
                if (instance == null) {          // 第二次检查（持锁）
                    instance = new Singleton();   // ← 出事的地方
                }
            }
        }
        return instance;
    }
}

去掉 volatile 会怎样？ instance = new Singleton() 不是原子操作，分三步：

1. memory = allocate(); // 分配内存
2. ctor(memory); // 调用构造函数初始化
3. instance = memory; // instance 指向内存

JIT/CPU 可能把它重排成 1 → 3 → 2（as-if-serial 在单线程看没问题）。于是：线程 A 执行到 3（instance 已非 null）但 2 还没做完——对象处于"半初始化"状态。此时线程 B 在第一次检查发现 instance != null，直接 return instance 拿去用，访问到一个字段全是默认值的残缺对象，可能 NPE 或脏数据。

volatile 怎么救：它在写 instance（步骤 3）前插 StoreStore 屏障，禁止 2、3 重排，保证"对象完全构造好"hb"instance 被赋值"hb"别的线程读到非 null"。这是一条标准的 happens-before 推导链。

检验 6. DCL 去掉 volatile 后最危险的后果是？

A. 创建出多个单例实例 B. 1→3→2 重排，别的线程读到"非 null 但未初始化完"的半成品对象 C. 死锁 D. synchronized 失效

7. final 字段的 JMM 保证 & synchronized 内存语义

final 字段的安全发布

JMM 对 final 字段有一条特殊保证：只要对象正确构造（构造期 this 不逃逸），构造函数内对 final 字段的写，对任何读到该对象引用的线程都可见——无需 volatile、无需锁。原理：JMM 在构造函数返回前插一道 StoreStore 屏障，保证 final 字段的写不会重排到构造函数外面。这就是 String、不可变对象天生线程安全的底层依据。

this 逃逸陷阱：如果构造函数还没结束就把 this 暴露出去（注册监听、塞进静态集合、启动引用 this 的线程），别的线程可能在 final 屏障生效之前拿到引用，读到 final 字段的默认值，破坏 final 保证。

public Foo() {
    this.x = 42;
    registry.add(this);   // ❌ this 逃逸！别的线程可能看到 x==0
}

synchronized 的内存语义：和 volatile 统一视角

很多人只把 synchronized 当"互斥"，其实它还有内存语义，对应 happens-before 的监视器锁规则：

  • 进入同步块（lock） ≈ volatile 读：清空本地缓存，从主存重新读共享变量。
  • 退出同步块（unlock） ≈ volatile 写：把同步块内的修改刷回主存。
  • 所以 unlock hb 后续同一锁的 lock，临界区内的所有写对下一个进入临界区的线程可见。
volatilesynchronized
可见性✅✅
有序性✅✅
原子性❌（单次读写除外）✅（整个临界区）
阻塞无锁，不阻塞会阻塞、可能上下文切换
适用状态标志、一写多读复合操作、需要互斥

跨语言对照（回贴 L15）：Go 没有 volatile，靠 channel 的发送 hb 接收、Mutex 的 Unlock hb 后续 Lock 建立 happens-before——思想和 Java 完全一致，都是"通过同步原语建立顺序保证"。

检验 7. 关于 final 字段的安全发布，正确的是？

A. final 字段必须再加 volatile 才能保证可见性 B. 只要构造期 this 不逃逸，final 字段的初始化对读到该对象的线程天然可见（构造尾部 StoreStore 屏障） C. final 字段也保证复合操作原子性 D. final 和 volatile 完全等价

8. 回贴：用 happens-before 重新解释 AQS 与 CHM

现在把理论闭环贴回前两课，这正是高级面试官想听的"知其所以然"：

  • AQS 的 volatile int state（L14）：线程 A 释放锁 setState(0) 是 volatile 写，线程 B 抢锁 getState() 是 volatile 读 → volatile 规则建立 hb，A 在临界区内的所有修改对 B 可见。这就是"锁能保护共享数据"的内存层根据，不只是互斥。
  • CHM 的 tabAt / setTabAt（L16）：table 元素的读用 Unsafe 的 acquire（volatile 读语义），写用 release（volatile 写语义）→ get 无锁也能看到别的线程刚 put 的桶头、以及扩容时的 ForwardingNode 标记。L16 当时说"靠 volatile 数组元素语义"，底层就是 happens-before。
  • clickhouse-mcp-server 实战：MCP server 多线程处理请求，配置热更新字段用 volatile（一写多读，靠 volatile 规则发布）；连接池的取/还是复合操作，靠 synchronized/锁的监视器规则发布。判断标准就是：这两个操作之间能不能连出 hb 链？连不上就有数据竞争。

面试话术

Q：volatile 能保证原子性吗？为什么？怎么解决？

"不能。volatile 只保证可见性和有序性，对单次读、单次写是原子的，但对 i++ 这种 read-modify-write 复合操作无能为力——两个线程可能同时读到旧值、各自加一、写回同一个值，丢更新。底层原因是 volatile 靠内存屏障让读写跨线程可见，但屏障管不了'读到改完再写'这三步之间的原子性。要解决得用 CAS，比如 AtomicInteger 的 incrementAndGet 内部自旋 CAS，或者直接 synchronized 把复合操作包成临界区。我在项目里区分得很清楚：停止标志位这种一写多读用 volatile，计数器用 LongAdder。"

Q：happens-before 有哪些规则？和 as-if-serial 什么关系？

"happens-before 是 JMM 的核心契约，含义是'A hb B 则 A 的结果对 B 可见且在 B 之前发生'，但它不要求按这个顺序执行，只要结果符合就允许重排。八条规则里最常用四条：程序顺序、监视器锁（unlock hb 同锁 lock）、volatile（写 hb 读）、传递性。as-if-serial 是更底层的约束——它只保证单线程内重排不改结果，多线程下就失效了。所以 happens-before 是建在 as-if-serial 之上、专门用来跨线程建立顺序保证的高层规则。判断并发 bug 我会去连 hb 链：关键读写之间连不出 hb，就是数据竞争。"

Q：DCL 单例为什么必须 volatile？不加会怎样？

"因为 instance = new Singleton() 不是原子的，分三步：分配内存、调构造函数、把引用赋给 instance。JIT/CPU 在单线程视角下允许把它重排成'分配→赋值→构造'。不加 volatile，线程 A 重排后 instance 已经非 null 但对象还没构造完，线程 B 在第一次检查时看到非 null 直接返回，拿到一个字段全是默认值的半成品对象。volatile 通过在写 instance 前插 StoreStore 屏障禁止这个重排，保证'对象构造完成 hb instance 赋值 hb 别的线程读到非 null'。这也是为什么 JDK 里 ConcurrentHashMap 的 table、AQS 的 state 都是 volatile——本质都是用 happens-before 做安全发布。"

Q8. 下面代码（a、b 普通变量，flag volatile）线程2 一定能看到 a==1, b==2 吗？

线程1: a=1; b=2; flag=true;   线程2: if(flag){ read a; read b; }
A. 不一定，普通变量没保证 B. 一定能：a、b 的写在 volatile 写之前，靠"程序顺序 + volatile + 传递性"hb 到线程2的读 C. 只能看到 a，看不到 b D. 取决于 CPU 型号

Q9. volatile 写之后插入的最强屏障 StoreLoad，作用是？

A. 防止后续普通写重排到 volatile 写之前 B. 确保 volatile 写刷出（store buffer flush），且不被后续的读越过——防写-读乱序 C. 让 volatile 变量变成原子自增 D. 释放锁

Q10. 线程 A 内：x=1; t.start();（t 内部读 x）。x 对 t 可见吗？

A. 不可见，x 不是 volatile B. 可见：线程启动规则——start() 之前的操作 hb 新线程内的任何操作 C. 只有 x 是 volatile 才可见 D. 需要 join 才可见

Q11. 关于 synchronized 的内存语义，正确的是？

A. 它只负责互斥，不管可见性 B. unlock hb 后续同一锁的 lock；进入≈volatile读，退出≈volatile写，临界区写对下一个进入者可见 C. 不同锁之间也能建立 hb D. synchronized 不保证原子性

Q12. Go 中如何建立跨 goroutine 的 happens-before？（回贴 L15）

A. 用 volatile 关键字 B. channel 的发送 hb 对应接收、Mutex 的 Unlock hb 后续 Lock——思想与 Java 一致 C. Go 没有内存模型，全靠运气 D. 只能用 atomic 包

课后首要行动

  1. 跑出可见性死循环：写一个 boolean running（不加 volatile）的工作线程 + 主线程置 false 的 demo，用 -server 模式跑，亲眼看它停不下来；加上 volatile 再跑一次，确认立即退出。
  2. 复现 i++ 丢更新：10 个线程对 volatile int count 各 count++ 10000 次，统计结果远小于 100000；换成 AtomicInteger 再跑，确认精确。
  3. 手画一条 hb 推导链：拿第 3 节的生产者-消费者例子，在纸上把 (1)hb(2)hb(3)hb(4) 四条边和用到的规则标全，再把 volatile 去掉，标出断在哪一环。
  4. 回标 AQS / CHM 源码：打开 JDK 的 AbstractQueuedSynchronizer 找到 volatile int state、ConcurrentHashMap 找到 volatile Node[] table 与 tabAt，在注释里写下"这里靠哪条 hb 规则做安全发布"。
  5. 项目落地一处 volatile：在 clickhouse-mcp-server 里给一个"配置热更新标志"或"优雅停机标志"加 volatile，并在 PR 描述里用 happens-before 一句话论证为什么够、为什么不需要锁。
